# Разбор `load_raw.py`

## Назначение

Этот документ подробно разбирает скрипт [`ingestion/load_raw.py`](/root/repos/db-bi-agent-bisup/ingestion/load_raw.py).
Его задача — объяснить, как проект загружает исходные CSV-файлы в слой `raw` в MinIO и почему этот шаг устроен именно так.

Это первый настоящий шаг конвейера данных после подготовки бакета и префиксов.
Он ещё не строит таблицы, не выполняет SQL и не делает бизнес-преобразования.
Здесь проект решает более базовую задачу: безопасно и повторяемо положить исходные файлы в lakehouse.

## Место скрипта в общем конвейере

Порядок работы проекта на верхнем уровне такой:

1. `scripts/init_storage.py` создаёт бакет и служебные префиксы.
2. `ingestion/load_raw.py` загружает исходные CSV в MinIO.
3. `ingestion/load_bronze.py` строит временные внешние таблицы и загружает `bronze`.
4. `ingestion/load_silver.py` формирует аналитическую модель `silver`.
5. `ingestion/load_marts.py` пересобирает витрины.

Из этого видно, что `load_raw.py` работает на границе между локальной файловой системой и объектным хранилищем.

## Какие файлы он загружает

В скрипте жёстко зафиксирован список исходных файлов:

- `articles.csv`
- `customers.csv`
- `transactions_train.csv`

Этот список хранится в константе `SOURCE_FILES`.
Скрипт не сканирует директорию произвольным образом и не пытается автоматически подхватывать новые файлы.

Это означает две вещи:

- поведение скрипта предсказуемо;
- при добавлении нового источника код придётся менять явно.

## Как устроен запуск

Разбор начинается с функции `parse_args()`.
Она настраивает два аргумента командной строки:

- `--load-date`
- `--source-dir`

### `--load-date`

Этот аргумент задаёт дату загрузки в формате `YYYY-MM-DD`.
Если его явно не передать, используется текущая дата:

```python
default=date.today().isoformat()
```

Это значение влияет не на содержимое файла, а на путь объекта в MinIO.

### `--source-dir`

Этот аргумент задаёт локальную директорию, из которой читаются CSV.
В коде его значение по умолчанию равно `settings.raw_prefix`.

Это важный нюанс.
В типовой конфигурации `RAW_PREFIX=raw`, а значит значение по умолчанию в самом скрипте будет `raw`, а не `data/raw`.

При этом в [`Makefile`](/root/repos/db-bi-agent-bisup/Makefile) для цели `load-raw` путь передаётся явно:

```bash
--source-dir data/raw
```

Поэтому фактический рекомендуемый сценарий использования проекта — запускать скрипт с явным `--source-dir`, а не рассчитывать на значение по умолчанию.

## Как создаётся клиент MinIO

Функция `build_minio_client()` собирает MinIO-клиент напрямую, без использования общего `S3Client`.
Это отдельная локальная реализация для конкретной задачи raw-загрузки.

### Какие параметры используются

Скрипт берёт из настроек:

- `settings.s3_endpoint`
- `settings.s3_access_key`
- `settings.s3_secret_key`

Дальше:

- проверяет, что значения не пустые;
- разбирает адрес через `urlparse`;
- определяет, нужен ли `https`;
- передаёт host и учётные данные в `Minio(...)`.

### Что важно заметить

В тексте ошибки список переменных назван как `MINIO_ENDPOINT`, `MINIO_ROOT_USER`, `MINIO_ROOT_PASSWORD`, но фактически значения берутся из `S3_ENDPOINT`, `S3_ACCESS_KEY`, `S3_SECRET_KEY`.
Это не ломает работу, но создаёт небольшое расхождение между названием сообщения и фактическими полями конфигурации.

Для понимания проекта это полезно помнить: пользовательский смысл здесь один и тот же, но код опирается именно на S3-настройки.

## Как считается MD5

Функция `compute_md5(file_path)` отвечает за вычисление контрольной суммы локального файла.

Логика простая:

- создаётся объект `hashlib.md5()`;
- файл открывается в бинарном режиме;
- содержимое читается чанками по `1 МБ`;
- каждый чанк добавляется в хеш;
- в конце возвращается итоговая строка MD5.

Такой подход важен по двум причинам:

- не нужно загружать весь файл целиком в память;
- появляется устойчивый способ понять, изменился файл или нет.

## Как читается MD5 уже загруженного объекта

Функция `get_object_md5(client, bucket, object_name)` пытается прочитать MD5 из метаданных объекта в MinIO.

Алгоритм:

1. вызвать `stat_object(...)`;
2. если объекта нет, вернуть `None`;
3. привести ключи метаданных к нижнему регистру;
4. попробовать найти MD5 в:
   - `x-amz-meta-md5`
   - `md5`

Это связано с тем, что пользовательские метаданные в S3-совместимом API могут приходить с префиксом `x-amz-meta-`.

## Как формируется путь объекта в MinIO

Ключевая логика загрузки находится в `upload_file(...)`.
Именно там строится имя объекта:

```python
object_name = f"{raw_prefix}/hm/{filename.split('.')[0]}/load_date={load_date}/{filename}"
```

Для каждого исходного файла это даёт такие пути:

- `raw/hm/articles/load_date=YYYY-MM-DD/articles.csv`
- `raw/hm/customers/load_date=YYYY-MM-DD/customers.csv`
- `raw/hm/transactions_train/load_date=YYYY-MM-DD/transactions_train.csv`

Это полностью согласовано с остальной документацией и с логикой `load_bronze.py`.

## Как устроена идемпотентность

Самая важная часть `load_raw.py` — это проверка, нужно ли вообще повторно загружать файл.

Алгоритм такой:

1. вычислить локальный MD5;
2. запросить MD5 уже лежащего в MinIO объекта;
3. если значения совпадают, пропустить загрузку;
4. если не совпадают, загрузить файл заново.

То есть идемпотентность здесь построена не на сравнении времён файлов и не на сравнении размеров, а именно на контрольной сумме содержимого.

Это хорошее решение для учебного проекта, потому что:

- оно легко объясняется;
- оно надёжнее сравнения по дате изменения;
- оно не требует отдельной служебной таблицы состояния загрузки.

## Что происходит при загрузке файла

Если удалённый MD5 не совпадает с локальным, вызывается:

```python
client.fput_object(
    bucket,
    object_name,
    file_path,
    content_type="text/csv",
    metadata={"md5": local_md5},
)
```

Здесь важно следующее:

- используется прямой метод загрузки файла из локального пути;
- явно задаётся `content_type="text/csv"`;
- MD5 записывается в пользовательские метаданные объекта.

То есть скрипт не просто кладёт файл в бакет, а сохраняет минимальную служебную информацию, нужную для повторного запуска.

## Как устроена функция `main()`

Функция `main()` собирает весь сценарий запуска.

Её шаги:

1. разобрать аргументы командной строки;
2. проверить, что передана директория с исходными файлами;
3. прочитать имя бакета и `raw_prefix` из настроек;
4. проверить, что обязательные значения конфигурации не пустые;
5. создать MinIO-клиент;
6. проверить существование бакета;
7. пройти циклом по `SOURCE_FILES`;
8. вызвать `upload_file(...)` для каждого файла.

Это простой и линейный сценарий.
В нём нет ветвящейся бизнес-логики, потому что у raw-загрузки задача именно техническая.

## Какие ошибки скрипт обрабатывает явно

### Нет директории источника

Если `source_dir` пустой, скрипт пишет ошибку в лог и завершает процесс через `sys.exit(1)`.

### Нет файла в директории

Если ожидаемый CSV отсутствует, `upload_file(...)` поднимает `FileNotFoundError`.

### Нет бакета

Если бакет ещё не создан, скрипт тоже завершает работу через `sys.exit(1)`.
Именно поэтому нормальный сценарий проекта предполагает, что `scripts/init_storage.py` был запущен заранее.

### Нет объекта в MinIO

Это не ошибка.
В `get_object_md5(...)` такая ситуация трактуется как нормальная: функция просто возвращает `None`, а скрипт выполняет загрузку.

## Что скрипт не делает

Это важно не меньше, чем то, что он делает.

`load_raw.py`:

- не создаёт бакет автоматически, если его нет;
- не строит внешние таблицы `hive.raw.*`;
- не делает SQL через Trino;
- не проверяет содержимое CSV на бизнес-корректность;
- не переименовывает поля;
- не приводит типы столбцов;
- не пишет данные в Iceberg.

Все эти задачи начинаются только на следующем шаге в `load_bronze.py`.

## Связь с остальным проектом

Связь со следующими уровнями очень жёсткая.

### Связь с `scripts/init_storage.py`

`load_raw.py` ожидает, что:

- бакет уже существует;
- префикс `raw/` уже подготовлен.

### Связь с `load_bronze.py`

`load_bronze.py` ищет файлы ровно по тому layout, который формирует `load_raw.py`.
Если поменять шаблон пути объектов здесь, следующий этап перестанет находить raw-данные.

### Связь с окружением

Скрипт зависит от:

- `APP_ENV`;
- MinIO/S3-параметров подключения;
- `LAKEHOUSE_BUCKET`;
- `RAW_PREFIX`.

То есть это не автономный утилитарный файл, а часть общей конфигурационной модели проекта.

## Ограничения текущей реализации

У текущей версии есть несколько практических ограничений.

### Жёстко зафиксирован набор файлов

Если появится новый источник, его нельзя просто положить в директорию и ожидать, что скрипт сам его подхватит.

### Значение `--source-dir` по умолчанию неудобно

В коде оно привязано к `settings.raw_prefix`, то есть обычно к строке `raw`.
Для локального запуска это менее удобно, чем `data/raw`, поэтому нормальный рабочий сценарий проекта опирается на явный аргумент в `Makefile`.

### Нет создания бакета внутри самого скрипта

Скрипт проверяет бакет, но не подготавливает всю структуру хранилища самостоятельно.
Это сознательное разделение ответственности, но при ручном запуске его легко забыть.

### Нет валидации формата CSV

Скрипт не проверяет заголовки, число столбцов и содержимое строк.
Он работает с файлами как с объектами, а не как с табличными структурами.

## Краткий итог

`load_raw.py` решает узкую, но очень важную задачу: перенести исходные CSV в объектное хранилище так, чтобы следующий этап мог стабильно их найти и прочитать.

Главные идеи этого скрипта:

- фиксированный layout raw-слоя;
- загрузка в MinIO через Python-клиент;
- идемпотентность по MD5;
- отсутствие бизнес-преобразований;
- жёсткая связка с последующим шагом `load_bronze.py`.

Если понимать этот файл, становится проще объяснить, почему в проекте raw — это именно слой файлов, а не сразу табличная модель.
