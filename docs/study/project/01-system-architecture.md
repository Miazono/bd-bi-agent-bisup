# Архитектура проекта

## Назначение

Этот документ нужен как учебная карта проекта `db-bi-agent-bisup`. Он объясняет, как в репозитории связаны объектное хранилище, каталог таблиц, SQL-движок, загрузочные скрипты и BI-контур, и чем отличается текущее состояние реализации от целевого.

## Что строит проект

Проект собирает локальный `lakehouse` поверх датасета `H&M Fashion Recommendations` и публикует поверх него BI-слой для SQL-доступа и вопросов на естественном языке. Базовая цепочка данных уже реализована как `raw -> bronze -> silver -> marts`, а BI-контур подключается поверх витрин.

В репозитории это выражено через несколько устойчивых зон ответственности:

- `docker-compose.yml` описывает локальные сервисы и их связи;
- `infra/` хранит конфиги сервисов;
- `ingestion/` содержит Python-скрипты оркестрации загрузки;
- `sql/ddl/` описывает физические таблицы;
- `sql/queries/` содержит SQL-преобразования;
- `bi-agent/` задает семантическую публикацию и артефакты BI-части.

## Главные компоненты

### MinIO

MinIO выступает объектным хранилищем для двух типов данных:

- исходные CSV-файлы слоя `raw`;
- физические данные Iceberg-таблиц слоев `bronze`, `silver` и `mart`.

В проекте используется бакет `lakehouse`. Логическая структура внутри бакета разделена на префиксы `raw/`, `bronze/`, `silver/` и `marts/`. Это не отдельные бакеты, а единая область хранения, разнесенная по смысловым каталогам.

### PostgreSQL

Отдельный контейнер PostgreSQL используется как внутренняя база метаданных Hive Metastore. Само содержимое lakehouse в PostgreSQL не хранится. В нём лежит только служебная информация каталога таблиц.

### Hive Metastore

Hive Metastore отвечает за метаданные Iceberg-таблиц. Через него Trino узнает, какие таблицы существуют, где лежат их файлы и как читать схемы. Для работы с MinIO Hive Metastore в образе дополнен `hadoop-aws` и `aws-java-sdk-bundle`, а параметры S3-совместимого доступа настраиваются через шаблонизацию конфигурации при старте контейнера.

### Trino

Trino в проекте играет роль SQL-движка. Он не хранит данные, а обращается к Metastore за метаданными и к MinIO за файлами. В конфигурации проекта у него два основных каталога:

- `hive` для временных внешних таблиц над raw-файлами;
- `iceberg` для физических таблиц `bronze`, `silver` и `mart`.

Это важное архитектурное разделение: один и тот же SQL-движок обслуживает и внешнее чтение CSV, и управляемые таблицы lakehouse.

### WrenAI

WrenAI вынесен в отдельный compose-профиль `wrenai`. Он не участвует в базовом конвейере загрузки и преобразования данных, а нужен как слой поверх Trino для семантического доступа и генерации SQL по естественному языку.

В текущем репозитории BI-контур уже подключен инфраструктурно, но его семантические артефакты заполнены неравномерно. Это означает, что сам стек уже собран, а содержательная часть BI-слоя еще доводится.

### Python

Python используется как оркестратор:

- `config/settings.py` загружает окружение и собирает единый объект настроек;
- `scripts/init_storage.py` подготавливает бакет и служебные префиксы;
- `ingestion/load_raw.py` кладет исходные CSV в MinIO;
- `ingestion/load_bronze.py` строит технический слой `bronze`;
- `ingestion/load_silver.py` строит аналитический слой `silver`;
- `ingestion/load_marts.py` пересобирает витрины `marts`.

При этом основная логика преобразования данных вынесена в SQL, а Python отвечает за порядок шагов, проверки и повторяемость запуска.

## Как сервисы связаны между собой

Связь в проекте построена по цепочке зависимостей:

1. Сначала поднимаются MinIO и PostgreSQL.
2. Затем стартует Hive Metastore, который использует PostgreSQL как метаданные и MinIO как хранилище файлов.
3. После этого запускается Trino, который подключается к Hive Metastore и читает данные из MinIO.
4. Затем выполняются Python-скрипты, которые через Trino и MinIO строят слои данных.
5. Поверх витрин может включаться WrenAI, который использует Trino как SQL-источник.

Физически это выражено в `docker-compose.yml`: сервисы запускаются в одной локальной среде и взаимодействуют по именам контейнеров и переменным окружения.

## Текущее состояние реализации

Сейчас в репозитории уже реализованы:

- локальный стек MinIO + PostgreSQL + Hive Metastore + Trino;
- загрузка `raw`, `bronze`, `silver` и `marts`;
- DDL и SQL для физической модели данных;
- базовый BI-контур на WrenAI;
- документация по слою данных и части инфраструктуры.

То есть проект уже не является только макетом: путь от файлов до аналитических таблиц действительно собран и может запускаться локально.

## Целевое состояние

Целевая архитектура проекта шире текущей реализации по BI-части. В идеале проект должен выглядеть так:

- `raw` хранит только исходные файлы без бизнес-преобразований;
- `bronze` остается близким к источнику и содержит технически нормализованные данные;
- `silver` содержит очищенные аналитические сущности;
- `marts` выступает главным слоем публикации для BI и семантического слоя;
- WrenAI строит SQL именно по витринам, а не по сырым таблицам;
- документация и семантические артефакты остаются синхронизированы с физической моделью.

Именно это целевое состояние уже зафиксировано в проектных документах и правилах репозитория, даже если отдельные части BI-слоя пока заполнены частично.

## Почему архитектура выбрана именно так

Такой стек хорошо подходит для учебного проекта по нескольким причинам:

- можно показать полный путь данных от файла до витрины;
- можно отдельно объяснить объектное хранилище, каталог метаданных и SQL-движок;
- можно продемонстрировать слоистую модель lakehouse без облачной инфраструктуры;
- можно построить BI-агента поверх стабильного SQL-слоя, а не поверх исходных файлов.

Для защиты это удобно тем, что каждый компонент выполняет свою роль и не дублирует функции других сервисов.
