# Разбор `load_marts.py`

## Назначение

Этот документ разбирает Python-скрипт [`ingestion/load_marts.py`](/root/repos/db-bi-agent-bisup/ingestion/load_marts.py), который отвечает за построение верхнего аналитического слоя `marts`.

Смысл разбора не только в том, чтобы понять последовательность вызовов, но и в том, чтобы увидеть, как Python здесь связывает:

- DDL физических витрин;
- SQL-запросы наполнения витрин;
- проверку структуры таблиц;
- полную пересборку готовых BI-таблиц;
- техническое логирование результата.

## Где этот скрипт находится в пайплайне

`load_marts.py` запускается после слоя `silver`.
Это не загрузка из raw и не техническая нормализация источника.
Здесь уже работают готовые аналитические сущности, а итогом становятся витрины, удобные для BI и семантического слоя.

Логически пайплайн выглядит так:

`raw` -> `bronze` -> `silver` -> `marts`

Именно `marts` являются верхним слоем публикации в текущем проекте.

## Общая идея скрипта

Скрипт устроен максимально прямо и предсказуемо:

1. создать схему `iceberg.mart`;
2. создать или подтвердить наличие физических таблиц витрин;
3. пройтись по всем витринам в фиксированном порядке;
4. для каждой витрины полностью удалить старые строки;
5. заново вставить данные из SQL-файла;
6. залогировать количество строк и сводку по файлам Iceberg.

Это не инкрементальный загрузчик и не сложный оркестратор.
Это именно скрипт полной пересборки витрин поверх уже подготовленного `silver`.

## Импорты и глобальные переменные

В начале файла подключаются стандартные и проектные зависимости:

- `logging` и `os` для журналирования;
- `Path` для работы с путями;
- `List` и `Tuple` для типизации;
- `settings` из `config/settings.py`;
- `SqlAssets` для поиска SQL-файлов;
- `TrinoClient` для выполнения SQL в Trino.

### Глобальные переменные

В модуле есть несколько констант:

- `LAKEHOUSE_BUCKET` — имя бакета;
- `MART_PREFIX` — префикс слоя витрин;
- `SQL_ASSETS` — объект для удобной навигации по SQL-файлам.

Это удобно, потому что скрипт дальше не дублирует строки путей и берёт настройки централизованно из `settings`.

## Логирование

Скрипт настраивает стандартное `logging.basicConfig(...)`.
Уровень логирования берётся из переменной окружения `LOG_LEVEL`, а формат включает время, уровень и сообщение.

Это важно по двум причинам:

- при запуске видно, на каком шаге сейчас находится пересборка;
- после каждой витрины можно быстро проверить, сколько строк и файлов получилось.

## Функция `mart_ddl_paths`

Функция `mart_ddl_paths()` возвращает список DDL-файлов для витрин.

Фактически она задаёт фиксированный порядок:

- `mart_sales_daily_channel.sql`;
- `mart_sales_monthly_category.sql`;
- `mart_customer_segment_monthly.sql`;
- `mart_repeat_purchase_category.sql`;
- `mart_customer_rfm_monthly.sql`.

Это не просто список файлов.
Это ещё и неявный порядок создания таблиц, который делает поведение скрипта стабильным и повторяемым.

## Функция `mart_query_specs`

Функция `mart_query_specs()` возвращает список пар:

- имя физической таблицы;
- путь к SQL-файлу наполнения.

Примеры:

- `iceberg.mart.sales_daily_channel` -> `sql/queries/mart/mart_sales_daily_channel.sql`;
- `iceberg.mart.customer_rfm_monthly` -> `sql/queries/mart/mart_customer_rfm_monthly.sql`.

Эта функция связывает физическую таблицу и её SQL-логику в одном месте.
Благодаря этому скрипт не ищет соответствие динамически и не хранит его в нескольких местах.

## Функция `execute_step`

`execute_step(trino, sql, title)` — небольшой обёрточный helper.

Он делает две вещи:

- пишет в лог сообщение `Start: ...`;
- выполняет SQL через `trino.execute(...)`;
- после завершения пишет `Done: ...`.

Такая обёртка нужна, чтобы любой шаг пайплайна был виден в журнале как отдельная операция.

## Функция `parse_fqn`

`parse_fqn(table_name)` разбирает fully qualified name вида `catalog.schema.table`.

В проекте это нужно не для красоты, а для доступа к Iceberg metadata tables.
Позже функция используется в `log_iceberg_files_summary`, где нужно сформировать запрос к `"$files"` конкретной таблицы.

Если имя таблицы задано не в формате из трёх частей, функция выбрасывает `ValueError`.
Это полезная защита от случайной передачи неполного имени.

## Функция `log_row_count`

`log_row_count(trino, table_name)` выполняет простой `SELECT COUNT(*)` и пишет число строк в лог.

Это базовая техническая проверка после пересборки витрины.
Она не доказывает бизнес-корректность, но показывает, что таблица не пустая и данные действительно записались.

## Функция `log_iceberg_files_summary`

Эта функция уже интереснее, потому что она работает не только со строками, но и с Iceberg metadata table.

### Что она делает

1. разбирает имя таблицы на `catalog`, `schema`, `table`;
2. обращается к системной таблице `"{table}$files"`;
3. считает сумму `record_count`, число файлов и общий размер в байтах;
4. пишет сводку в лог.

### Зачем это нужно

Для Iceberg важно понимать не только количество строк, но и физическую упаковку данных.
Если таблица пересобралась, но файлов нет, это тревожный сигнал.
Если файлов слишком много, это тоже может быть признаком неудачной физической структуры.

Поэтому эта проверка полезна как очень лёгкий слой контроля после записи.

## Функция `ensure_mart_schema`

Эта функция создаёт схему `iceberg.mart`, если её ещё нет.

Она строит location на основе:

- схемы таблиц `s3a`;
- имени бакета `lakehouse`;
- префикса `marts/`.

Фактически это означает, что все витрины будут храниться в отдельном физическом месте внутри MinIO, но в рамках того же lakehouse-стека.

### Почему это важно

Если схема не создана заранее, DDL витрин может не выполниться.
Кроме того, явное создание схемы делает скрипт самодостаточным: он не зависит от ручной подготовки каталога перед запуском.

## Функция `ensure_mart_tables`

Эта функция проходит по DDL-файлам витрин и вызывает `trino.execute_file(...)` для каждого файла.

Здесь принцип простой:

- файл есть — выполняем;
- файла нет — сразу ошибка `FileNotFoundError`.

Такой подход удобен для учебного проекта, потому что ошибка в структуре репозитория видна сразу.
Скрипт не пытается “догадаться”, где лежит таблица и как её создать.

## Функция `rebuild_mart_table`

Это центральная функция всего скрипта.

### Что она делает

1. проверяет, что SQL-файл наполнения существует;
2. читает текст SQL из файла;
3. удаляет все строки из целевой таблицы через `DELETE FROM`;
4. выполняет `INSERT INTO ...` из SQL-файла.

### Почему SQL читается как текст

Функция не пытается парсить SQL и не разделяет его на шаги.
Она просто берёт готовый запрос из файла и выполняет его целиком.

Это хороший подход для слоя `marts`, потому что:

- логика агрегации живёт в отдельном SQL-файле;
- Python не дублирует бизнес-логику;
- SQL проще ревьюить отдельно;
- поведение пересборки остаётся прозрачным.

### Почему сначала `DELETE`, а потом `INSERT`

Это сознательно выбранная стратегия полной пересборки витрины.
Она проще, чем частичный `MERGE`, потому что:

- не нужно вычислять изменившиеся ключи;
- не нужно поддерживать сложную логику слияния;
- каждый запуск приводит витрину к одному и тому же состоянию.

Для учебного проекта это особенно удобно: результат предсказуем, а реализация легче объясняется на защите.

## Функция `main`

`main()` собирает весь сценарий запуска.

### Последовательность шагов

1. создаётся объект `TrinoClient`;
2. создаётся схема `iceberg.mart`;
3. применяются DDL-файлы витрин;
4. по каждой витрине выполняется полная пересборка;
5. после каждой пересборки пишется количество строк;
6. затем пишется Iceberg-сводка по файлам;
7. в конце логируется завершение загрузки.

### Важная особенность

У `load_marts.py` нет `argparse` и пользовательских CLI-аргументов.
Это значит, что сценарий запуска здесь намеренно фиксированный.

Скрипт не ждёт от пользователя выбора витрин, периода или режима обновления.
Он всегда пересобирает весь набор витрин в одном и том же порядке.

## Какие предпосылки нужны для работы скрипта

Чтобы `load_marts.py` отработал успешно, должны быть выполнены несколько условий:

- уже должны существовать и быть заполнены таблицы `silver`;
- должен быть доступен Trino;
- должны существовать SQL-файлы в `sql/ddl/mart/` и `sql/queries/mart/`;
- должны быть корректно заданы настройки из `config/settings.py`;
- в объектном хранилище должен быть доступен бакет `lakehouse`;
- схема `iceberg.mart` должна быть доступна на запись.

Если хотя бы один из этих компонентов не готов, пересборка витрин не получится.

## Связь с другими частями проекта

Этот скрипт работает не сам по себе.
Он опирается на несколько других уровней проекта:

- `config/settings.py` даёт окружение и адреса сервисов;
- `SqlAssets` помогает найти DDL и SQL-файлы;
- `TrinoClient` выполняет SQL;
- `sql/ddl/mart/` задаёт физическую структуру витрин;
- `sql/queries/mart/` задаёт расчёт метрик и грейн каждой витрины;
- `silver` уже должен содержать подготовленные данные.

Именно поэтому `load_marts.py` лучше рассматривать как финальную точку конвейера, а не как самостоятельный ETL-скрипт.

## Ограничения подхода

У полной пересборки витрин есть несколько ограничений.

### Нет инкрементальности

Скрипт не обновляет только изменившуюся часть данных.
Он всегда очищает витрину целиком и строит её заново.

Это упрощает код, но может быть медленнее на больших объёмах.

### Нет частичного восстановления

Если сбой произошёл посередине пересборки, витрина может остаться пустой или частично очищенной до повторного запуска.

В учебном проекте это обычно приемлемо, но в промышленной системе такую схему часто усиливают дополнительной атомарностью или временными таблицами.

### Нет отдельной бизнес-ветвки в Python

Python здесь не решает, какую метрику считать или как группировать данные.
Вся аналитическая логика уже заранее зашита в SQL-витринах.

Это делает скрипт проще, но означает, что качество результата зависит от корректности SQL-файлов.

### Полная пересборка дороже по времени

Если витрины станут крупнее, `DELETE` + `INSERT` может стать заметно тяжелее, чем точечное обновление.
Сейчас это компенсируется малым учебным объёмом данных и предсказуемостью реализации.

## Что полезно помнить про этот скрипт

- Он не содержит пользовательских параметров запуска.
- Он пересобирает витрины полностью, а не частично.
- Он не вычисляет метрики сам, а только исполняет SQL.
- Он даёт техническую проверку через количество строк и Iceberg metadata table.
- Он является финальным шагом верхнего аналитического слоя.

## Итог

`load_marts.py` — это аккуратный и очень прямой оркестратор верхнего слоя `marts`.
Его сила в простоте: схема создаётся явно, таблицы проверяются явно, витрины пересобираются явно, результат логируется явно.

Для защиты проекта это хороший скрипт, потому что его легко объяснить как финальную точку цепочки:

`silver` -> `marts` -> BI-экспозиция

При этом важно честно говорить о компромиссе: скрипт удобен и прозрачен, но работает по схеме полной пересборки, а значит не является оптимальным инкрементальным решением для очень больших данных.
