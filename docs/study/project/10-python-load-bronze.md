# Разбор `load_bronze.py`

## Зачем этот документ

Этот материал разбирает Python-скрипт, который переводит данные из `raw` в `bronze`.
Здесь важно понять не только порядок вызовов, но и то, как скрипт связывает MinIO, временные внешние таблицы `hive.raw.*`, DDL-файлы и SQL-запросы слоя `bronze`.

Разбор опирается только на фактическую реализацию:

- [`ingestion/load_bronze.py`](/root/repos/db-bi-agent-bisup/ingestion/load_bronze.py)
- [`ingestion/utils/s3_client.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/s3_client.py)
- [`ingestion/utils/trino_client.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/trino_client.py)
- [`ingestion/utils/sql_assets.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/sql_assets.py)
- [`ingestion/utils/sql_templates.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/sql_templates.py)
- [`sql/ddl/bronze/`](/root/repos/db-bi-agent-bisup/sql/ddl/bronze)
- [`sql/queries/bronze/`](/root/repos/db-bi-agent-bisup/sql/queries/bronze)

## Общая роль скрипта

`load_bronze.py` отвечает за первый табличный этап обработки данных.
Он не делает сложную бизнес-логику, а решает более прикладную задачу:

- проверить, что raw-файлы действительно лежат в MinIO;
- создать или переиспользовать нужные схемы;
- создать временные внешние таблицы над CSV;
- создать физические таблицы `bronze`;
- очистить старые строки текущего `batch_id`;
- загрузить новые строки в `iceberg.bronze.*`;
- вывести базовую статистику по результату.

Это делает скрипт оркестратором слоя `bronze`, а не просто набором SQL-вызовов.

## Импорты и зависимости

В скрипте используются:

- стандартные модули `argparse`, `logging`, `os`, `Path`, `Dict`, `List`;
- `S3Client` для работы с MinIO;
- `TrinoClient` для выполнения SQL;
- `SqlAssets` для поиска и рендера SQL-файлов;
- `settings` для доступа к переменным окружения.

Здесь важно разделение ответственности:

- MinIO нужен для проверки и чтения файлов;
- Trino нужен для создания схем, внешних таблиц и физических таблиц;
- SQL-файлы хранят трансформационную логику отдельно от Python;
- Python управляет порядком шагов и безопасностью повторного запуска.

## Глобальные константы

В верхней части файла задаются:

- `LAKEHOUSE_BUCKET` — имя бакета;
- `RAW_PREFIX` — префикс слоя `raw`;
- `BRONZE_PREFIX` — префикс слоя `bronze`;
- `SQL_ASSETS` — объект для доступа к SQL-артефактам.

Эти значения берутся из [`config/settings.py`](/root/repos/db-bi-agent-bisup/config/settings.py), поэтому поведение скрипта зависит от окружения `local` или `docker`.

## `RAW_LAYOUT`

`RAW_LAYOUT` описывает ожидаемую физическую структуру raw-файлов в MinIO.

В словаре хранятся три источника:

- `articles`;
- `customers`;
- `transactions_train`.

Для каждого источника задан относительный путь с `load_date`:

- `hm/articles/load_date={load_date}/articles.csv`;
- `hm/customers/load_date={load_date}/customers.csv`;
- `hm/transactions_train/load_date={load_date}/transactions_train.csv`.

Это важная часть контракта загрузки.
Скрипт не ищет файлы “где-то в бакете”, а ожидает строго определённую структуру.

## `RAW_TABLE_COLUMNS`

`RAW_TABLE_COLUMNS` задаёт колонки временных внешних таблиц `hive.raw.*`.

Здесь важно несколько вещей.

### `hm_articles_raw`

Колонки полностью повторяют структуру `articles.csv`:

- идентификаторы товара и производных кодов;
- названия категорий и групп;
- признаки внешнего вида, цвета и отделов;
- описание товара.

### `hm_customers_raw`

Колонки повторяют `customers.csv`:

- `customer_id`;
- `fn`;
- `active`;
- `club_member_status`;
- `fashion_news_frequency`;
- `age`;
- `postal_code`.

### `hm_transactions_raw`

Колонки повторяют `transactions_train.csv`:

- `t_dat`;
- `customer_id`;
- `article_id`;
- `price`;
- `sales_channel_id`.

Смысл этого словаря в том, чтобы временная внешняя таблица читала CSV как плоский текстовый источник, а не как уже типизированную модель.

## Вспомогательные функции

### `q`

Функция `q(value)` экранирует одинарные кавычки для подстановки строки в SQL.
Она используется там, где в запросы вставляются `batch_id` и имя файла.

Это простая, но важная защита от поломки SQL-строки.

### `bronze_ddl_paths`

Функция возвращает список путей к DDL-файлам `bronze`:

- `bronze_hm_articles.sql`;
- `bronze_hm_customers.sql`;
- `bronze_hm_transactions.sql`.

### `bronze_query_paths`

Функция возвращает карту путей к SQL-запросам слоя `bronze`.

В текущей реализации она существует как удобный справочник, но в `main()` не используется напрямую.
Фактический запуск обращается к SQL-файлам через `SqlAssets.render(...)`.

Это хороший пример того, как в коде может быть подготовлена вспомогательная обвязка, но реальный поток выполнения идёт другим путём.

## Проверка raw-файлов

### `validate_raw_files`

Эта функция проверяет, что все нужные raw-файлы реально присутствуют в MinIO.

Она проходит по `RAW_LAYOUT`, собирает ключ для каждого файла и проверяет его через `s3.object_exists(...)`.
Если хотя бы одного файла нет, скрипт падает с `FileNotFoundError`.

### Зачем это нужно

Это предотвращает ситуацию, когда пайплайн начнёт строить `bronze` на частично загруженных данных.
Проверка делается до создания таблиц и до вставки строк, то есть сбой происходит рано и явно.

### Что возвращает функция

Функция возвращает словарь `resolved`, где для каждого источника хранится полный ключ объекта в MinIO.
Этот словарь потом используется при создании временных внешних таблиц.

## Создание схем

### `create_schemas`

Функция создаёт две схемы:

- `hive.raw`;
- `iceberg.bronze`.

Схема `hive.raw` нужна для временных внешних таблиц над CSV.
Схема `iceberg.bronze` нужна для физических таблиц слоя `bronze`.

### Почему схема создаётся в Python

Это делает загрузчик самодостаточным:

- не нужно вручную подготавливать схему перед запуском;
- повторный запуск не ломается, если схема уже существует;
- оркестрация остаётся в одном месте.

## Временные внешние таблицы

### `create_raw_external_table`

Эта функция создаёт одну внешнюю таблицу в `hive.raw`.

Сначала старое определение удаляется через `DROP TABLE IF EXISTS`, потом создаётся новая таблица:

- с набором столбцов из `RAW_TABLE_COLUMNS`;
- с `VARCHAR`-типом для всех колонок;
- с `external_location`, указывающим на каталог файлов;
- с `format = 'CSV'`;
- с `skip_header_line_count = 1`.

### Почему все столбцы сначала `VARCHAR`

Это сознательный технический приём.
Сначала CSV читается как текстовый набор колонок, а уже потом значения приводятся к нужным типам в запросах `bronze`.

Такой подход удобен, потому что:

- минимизирует риск падения на этапе чтения файла;
- переносит приведение типов в явный SQL;
- делает логику загрузки более прозрачной.

### `create_all_raw_tables`

Эта функция вызывает `create_raw_external_table` три раза:

- для `hm_articles_raw`;
- для `hm_customers_raw`;
- для `hm_transactions_raw`.

Для каждой таблицы берётся каталог файла через `Path(...).parent`, а URI собирается через `s3.build_dir_uri(...)`.

## Создание физических таблиц bronze

### `create_bronze_tables`

Функция проходит по DDL-файлам из `bronze_ddl_paths()` и выполняет каждый через `trino.execute_file(...)`.

Это означает, что физическая структура таблиц задаётся отдельно от логики загрузки:

- DDL создаёт таблицы;
- SQL-запросы наполняют их данными;
- Python только связывает оба этапа в один запуск.

## Очистка предыдущего batch

### `delete_batch`

Перед загрузкой нового batch скрипт удаляет строки с тем же `batch_id` из трёх бронзовых таблиц:

- `iceberg.bronze.hm_articles`;
- `iceberg.bronze.hm_customers`;
- `iceberg.bronze.hm_transactions`.

### Зачем это нужно

Такой шаг делает загрузку идемпотентной:

- повторный запуск не накопит дубликаты;
- данные одного batch можно безопасно пересобрать;
- если batch запускают повторно, старый результат сначала убирается.

## Загрузка в bronze

### `load_articles_to_bronze`

Функция рендерит `sql/queries/bronze/load_hm_articles.sql` и подставляет:

- `SOURCE_FILE_NAME`;
- `BATCH_ID`.

После этого SQL выполняется через Trino.

### `load_customers_to_bronze`

Функция делает то же самое для `load_hm_customers.sql`.

### `load_transactions_to_bronze`

Функция делает то же самое для `load_hm_transactions.sql`.

### Как это связано с SQL-слоем

Внутри самих SQL-файлов выполняются:

- приведение типов;
- нормализация пустых значений;
- добавление `ingest_ts`;
- запись имени файла и `batch_id`.

Python здесь не повторяет эту логику, а только подставляет параметры и запускает запрос.

## Логирование результата

### `log_counts`

После загрузки скрипт считает количество строк в каждой бронзовой таблице по текущему `batch_id`.

Это простая проверка, которая помогает быстро увидеть, что данные действительно попали в таблицы.

## Аргументы запуска

### `--load-date`

Обязательный параметр.
Используется для выбора нужной папки `load_date=...` в raw.

### `--batch-id`

Обязательный параметр.
Служит идентификатором загрузки и используется:

- в `DELETE` старых строк;
- в SQL-запросах наполнения;
- в дальнейшем обновлении `silver`.

### `--raw-prefix`

Необязательный параметр.
По умолчанию берётся из `RAW_PREFIX` в окружении.
Позволяет явно переопределить корневой префикс raw-файлов.

## Полный порядок выполнения

Если собрать `main()` по шагам, конвейер выглядит так:

1. создаётся `S3Client` и `TrinoClient`;
2. проверяется, что бакет существует, либо создаётся;
3. проверяются все raw-файлы для указанной даты;
4. создаются схемы `hive.raw` и `iceberg.bronze`;
5. создаются временные внешние таблицы `hive.raw.*`;
6. создаются физические таблицы `bronze`;
7. удаляются строки с тем же `batch_id`;
8. данные загружаются в `bronze` из трёх SQL-файлов;
9. выводятся счётчики строк;
10. скрипт завершается успешным сообщением в лог.

## Ограничения реализации

У текущего `load_bronze.py` есть сознательные ограничения.

- Он работает только с тремя исходными файлами.
- Он ожидает строго заданную структуру каталогов в MinIO.
- Он не делает сложную бизнес-очистку, а только техническую нормализацию.
- Он не управляет семантическим слоем и не строит витрины.
- Он создаёт временные внешние таблицы заново при каждом запуске.

Это нормальная граница для бронзового уровня: его задача не аналитика, а стабильная и прозрачная подготовка данных к следующему слою.

## Что важно помнить

Для защиты проекта полезно уметь объяснить три вещи:

- `load_bronze.py` не просто копирует файлы, а связывает raw-слой, внешние таблицы и физический `bronze`;
- типы данных приводятся не в Python, а в SQL-запросах слоя `bronze`;
- повторный запуск защищён через проверку raw-файлов и удаление старого `batch_id`.

Именно поэтому этот скрипт — ключевой мост между объектным хранением и аналитическими таблицами.
