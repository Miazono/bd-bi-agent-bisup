# Разбор DDL и физической модели

## Назначение

Этот документ объясняет, как в проекте устроена физическая модель таблиц и как читать DDL-файлы из каталога `sql/ddl/`.
Его задача не просто перечислить поля, а показать, какие решения приняты на уровне хранения, типов и партиционирования.

## Что такое DDL в этом проекте

DDL в проекте отвечает за физические определения таблиц:

- имя таблицы;
- набор столбцов и их типы;
- формат хранения;
- физический путь в объектном хранилище;
- правила партиционирования.

В проекте DDL-файлы лежат в:

- `sql/ddl/bronze/`;
- `sql/ddl/silver/`;
- `sql/ddl/mart/`.

Каждый файл описывает одну физическую таблицу Iceberg.

## Общий шаблон DDL

Почти все DDL в проекте устроены одинаково:

```sql
CREATE TABLE IF NOT EXISTS <schema>.<table> (
  ...
)
WITH (
  format = 'PARQUET',
  location = 's3a://lakehouse/.../',
  partitioning = ARRAY['month(...)']
);
```

Из этого шаблона нужно понимать несколько вещей.

### `CREATE TABLE IF NOT EXISTS`

Эта конструкция говорит, что таблица создаётся только если её ещё нет.
Повторный запуск DDL не должен ломать пайплайн, если структура уже создана.

### `<schema>.<table>`

В DDL используются схемы:

- `bronze`;
- `silver`;
- `mart`.

Это имена схем внутри каталога `iceberg`, то есть фактически речь идёт о таблицах вида:

- `iceberg.bronze.hm_articles`;
- `iceberg.silver.dim_article`;
- `iceberg.mart.sales_daily_channel`.

В самих DDL каталог `iceberg` не прописан, потому что DDL исполняются в контексте нужного подключения Trino.

### Список столбцов

Список столбцов задаёт физическую структуру таблицы:

- имя поля;
- тип данных;
- порядок полей.

В проекте все поля описываются явно.
Автоматического вывода схемы из CSV или из `SELECT` здесь нет.

### `WITH (format = 'PARQUET')`

Все физические таблицы проекта хранятся в формате `PARQUET`.
Это значит:

- данные лежат в колонночном формате;
- таблицы ориентированы на аналитическое чтение;
- один и тот же формат используется в `bronze`, `silver` и `mart`.

### `location = 's3a://...'`

Каждая таблица явно привязана к физическому пути в MinIO через `s3a://`.
Это важно по двум причинам:

- можно точно понимать, где лежат файлы таблицы;
- схема хранения согласована со слоями данных.

Например:

- `s3a://lakehouse/bronze/hm_articles/`;
- `s3a://lakehouse/silver/dim_article/`;
- `s3a://lakehouse/marts/sales_daily_channel/`.

### `partitioning = ARRAY[...]`

Эта секция задаёт физическое партиционирование таблицы.
Партиционирование есть не у всех таблиц, а только там, где это даёт практический выигрыш.

В проекте используется партиционирование по месяцу даты:

- `month(t_dat)`;
- `month(sale_date)`;
- `month(sale_month)`;
- `month(snapshot_month)`.

## Как устроены DDL по слоям

## Bronze

### Общая логика

DDL слоя `bronze` описывают технически нормализованные таблицы, близкие к исходным файлам.
Поэтому у них есть две характерные черты:

- почти все бизнес-поля похожи на поля источника;
- в каждую таблицу добавлены технические поля загрузки.

### Общие технические поля

Во всех таблицах `bronze` есть:

- `ingest_ts` — момент вставки строки;
- `source_file_name` — имя файла-источника;
- `batch_id` — идентификатор загрузки.

Эти поля не относятся к предметной области H&M.
Они нужны для повторных прогонов, трассировки и инкрементального обновления следующих слоёв.

### `bronze.hm_articles`

Таблица хранит товарный справочник в почти исходной форме.
Типы здесь уже приведены:

- идентификаторы и числовые коды в `BIGINT` или `INTEGER`;
- текстовые поля в `VARCHAR`.

Партиционирования нет.
Это разумно, потому что таблица относительно компактная и по ней не нужен отдельный месячный доступ.

### `bronze.hm_customers`

Таблица хранит клиентский справочник.
Здесь тоже нет партиционирования.
Типы выбраны так, чтобы дальше было удобно строить `dim_customer`:

- `customer_id` и текстовые статусы остаются `VARCHAR`;
- числовые признаки `fn`, `active`, `age` переводятся в `INTEGER`.

### `bronze.hm_transactions`

Это самая важная таблица слоя `bronze`, потому что именно она становится основой фактов и витрин.

Ключевые особенности:

- `t_dat` хранится как `DATE`;
- `price` хранится как `DECIMAL(12,4)`;
- таблица партиционируется по `month(t_dat)`.

Партиционирование здесь особенно важно, потому что:

- транзакции являются самым крупным набором данных проекта;
- `silver.fact_sales_line` пересчитывается по затронутым месяцам;
- витрины тоже во многом зависят от временной структуры продаж.

## Silver

### Общая логика

DDL слоя `silver` описывают очищенную аналитическую модель.
По сравнению с `bronze` здесь видно более предметное устройство:

- появляются измерения;
- появляется факт продаж;
- появляются производные признаки;
- появляется отдельный агрегат по паре клиент - товар.

### `silver.dim_article`

Эта таблица почти повторяет структуру товарного справочника из `bronze`, но дополняется аналитическими признаками:

- `is_ladieswear`;
- `is_menswear`;
- `is_kids`;
- `color_family`.

Типы этих полей уже отражают смысл:

- флаги хранятся как `BOOLEAN`;
- `color_family` как `VARCHAR`.

### `silver.dim_customer`

В размерности клиента появляются производные поля:

- `age_band`;
- `is_active_customer`;
- `is_fn_flag_present`.

Здесь хорошо видно различие между техническим и аналитическим слоем:

- в `bronze` хранились более сырые значения `active` и `fn`;
- в `silver` появляются аналитически удобные булевы признаки и возрастной диапазон.

### `silver.dim_date`

Это календарное измерение.
Оно хранит только календарные признаки, без технических полей загрузки.

Состав полей небольшой:

- дата;
- год;
- месяц;
- день месяца;
- день недели;
- номер недели.

Такой DDL показывает, что таблица имеет строго вспомогательное аналитическое назначение.

### `silver.fact_sales_line`

Это основной факт продаж.
По структуре он близок к `bronze.hm_transactions`, но дата уже переименована в `sale_date`, а сама таблица формально входит в аналитический слой.

Ключевые особенности:

- сохраняются технические поля загрузки;
- используется `DECIMAL(12,4)` для цены;
- задано партиционирование по `month(sale_date)`.

Это партиционирование согласовано с логикой загрузчика `load_silver.py`, который умеет обновлять факт по месячным частям.

### `silver.fact_customer_article_stats`

Это производный агрегат.
Здесь уже нет технических полей загрузки и нет даты события в виде одной строки продажи.
Вместо этого таблица хранит итоговую статистику по паре:

- `customer_id`;
- `article_id`.

Поля таблицы:

- `first_purchase_date`;
- `last_purchase_date`;
- `purchase_cnt`;
- `total_revenue`;
- `avg_price`.

По DDL уже видно, что это не транзакционный факт, а агрегированная аналитическая сущность.

## Mart

### Общая логика

DDL слоя `mart` описывают витрины, предназначенные для BI и семантической публикации.
У них есть несколько общих признаков:

- в структуре почти нет сырых технических полей;
- поля названы как готовые аналитические измерения и метрики;
- часть витрин партиционируется по месяцу или дню, а часть нет.

### `mart.sales_daily_channel`

Витрина хранит дневные продажи по каналу.
Столбцы уже ориентированы на готовую аналитику:

- измерения: `sale_date`, `sales_channel_id`;
- метрики: `revenue`, `items_sold`, `buyers_cnt`, `avg_item_price`.

Партиционирование по `month(sale_date)` помогает для временного анализа.

### `mart.sales_monthly_category`

Витрина хранит месячные продажи по категории.
Здесь видно, что поле категории заранее материализовано в витрину, а не требует соединения на стороне BI.

Поля:

- `sale_month`;
- `category`;
- `revenue`;
- `items_sold`;
- `buyers_cnt`;
- `active_sku_cnt`.

### `mart.customer_segment_monthly`

Это витрина по сегментам клиентов на месячном уровне.
В DDL уже видно, что сегмент хранится как готовое измерение `customer_segment`, хотя его смысл определяется SQL-логикой наполнения.

### `mart.repeat_purchase_category`

Это витрина без временного партиционирования.
Причина понятна по структуре: одна строка представляет одну категорию и итоговые показатели повторной покупки по ней.

### `mart.customer_rfm_monthly`

Это наиболее аналитически насыщенная витрина слоя `mart`.
Она хранит:

- клиента;
- месяц снимка;
- давность последней покупки;
- частоту;
- денежную метрику;
- сегмент RFM.

Партиционирование по `month(snapshot_month)` соответствует логике ежемесячных снимков.

## Чего нет в DDL

Это не менее важно, чем то, что в DDL есть.

### Нет `PRIMARY KEY`

В проекте не задаются физические первичные ключи.
Это означает, что уникальность и грейн поддерживаются логикой загрузки, а не ограничениями на уровне таблиц.

### Нет `FOREIGN KEY`

Внешние ключи тоже не задаются.
Связи между таблицами существуют логически:

- `fact_sales_line.customer_id` соотносится с `dim_customer.customer_id`;
- `fact_sales_line.article_id` соотносится с `dim_article.article_id`;
- `fact_sales_line.sale_date` соотносится с `dim_date.date_day`.

Но физически эти связи не enforced средствами таблиц.

### Нет скрытой бизнес-логики

DDL не отвечает на вопрос, как вычисляются производные поля и метрики.
Например:

- DDL показывает, что поле `is_ladieswear` существует;
- но только SQL-запрос `refresh_dim_article_insert.sql` показывает, как именно оно вычисляется.

Поэтому DDL нужно читать вместе с `sql/queries/`.

## Как читать DDL вместе с остальными файлами

Полезный порядок чтения такой:

1. Сначала определить назначение таблицы и её грейн.
2. Затем посмотреть DDL, чтобы понять структуру, типы и партиционирование.
3. Потом открыть SQL-запрос наполнения и понять, откуда берётся каждая колонка.
4. После этого свериться с Python-лоадером, чтобы понять стратегию обновления.

Именно в таком порядке физическая модель начинает восприниматься как часть пайплайна, а не как набор отдельных файлов.

## Итог

DDL проекта устроены достаточно последовательно и хорошо отражают слоистую модель данных:

- `bronze` хранит технически нормализованные структуры;
- `silver` хранит аналитические сущности;
- `mart` хранит витрины для BI.

Главное, что нужно вынести из этих файлов: DDL в проекте описывает не бизнес-логику, а физический каркас данных.
Чтобы действительно понять поведение таблиц, DDL обязательно нужно читать вместе с SQL-запросами и загрузочными скриптами.
