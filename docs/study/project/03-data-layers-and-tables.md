# Слои данных и таблицы проекта

## Зачем этот документ

Этот материал нужен, чтобы связать воедино три вещи:

- физическое хранение данных в проекте;
- логические слои модели `raw -> bronze -> silver -> marts`;
- реальные таблицы, запросы и правила наполнения, которые уже реализованы в репозитории.

Здесь нет общей теории “в вакууме”. Описывается только то, что реально есть в проекте `db-bi-agent-bisup`.

## Как читать модель

В проекте важно разделять логический и физический уровни.

- `raw` - это слой исходных файлов в MinIO.
- `bronze`, `silver` и `mart` - это физические таблицы, которые читаются и наполняются через Trino.
- `marts` - это логическое название верхнего аналитического слоя.
- `mart` - это физическая схема, в которой лежат витрины.

Из этого следует важный вывод: когда в документации говорится “слой marts”, речь идёт о бизнес-уровне публикации данных, а когда говорится `iceberg.mart`, это уже конкретное место хранения.

## Сводная цепочка

Общий путь данных в проекте выглядит так:

`CSV-файлы` -> `raw` в MinIO -> временные внешние таблицы `hive.raw.*` -> таблицы `bronze` -> таблицы `silver` -> витрины `mart`

При этом:

- `raw` хранит исходные файлы без бизнес-преобразований;
- `bronze` максимально близок к источнику, но уже содержит технические поля и нормализацию типов;
- `silver` формирует очищенную аналитическую модель;
- `marts` дают удобный для BI-агента и аналитики уровень агрегатов.

## Raw

### Что это такое

`raw` в проекте - это не таблицы, а исходные CSV-файлы, разложенные по бакетам и префиксам в MinIO:

- `raw/hm/articles/load_date=YYYY-MM-DD/articles.csv`;
- `raw/hm/customers/load_date=YYYY-MM-DD/customers.csv`;
- `raw/hm/transactions_train/load_date=YYYY-MM-DD/transactions_train.csv`.

Это слой хранения исходника без бизнес-логики.

### Что здесь важно

- В `raw` сохраняется исходное содержимое файлов.
- Имена бизнес-полей не меняются.
- Файлы версионируются по `load_date`.
- При повторной загрузке тот же файл не сохраняется повторно, если его MD5 не изменился.

### Временные внешние таблицы

Для чтения `raw` в процессе загрузки bronze создаются временные внешние таблицы в схеме `hive.raw`:

- `hive.raw.hm_articles_raw`;
- `hive.raw.hm_customers_raw`;
- `hive.raw.hm_transactions_raw`.

Они нужны только как технический мост между файлами в MinIO и таблицами `bronze`. Это не самостоятельный аналитический слой.

## Bronze

### Общая роль слоя

`bronze` - это технически нормализованный слой. Он уже хранится как таблицы и уже пригоден для SQL-обработки, но по смыслу остаётся близким к источнику.

На этом слое допустимы:

- приведение типов;
- нормализация пустых значений;
- добавление технических полей загрузки;
- минимальная техническая нормализация без тяжёлой бизнес-логики.

Тяжёлую аналитику и агрегации сюда не выносят.

### `bronze.hm_articles`

- Источник: `hive.raw.hm_articles_raw`.
- Grain: 1 строка = 1 товарная запись.
- Назначение: технический справочник товаров, максимально близкий к источнику.
- Технические поля: `ingest_ts`, `source_file_name`, `batch_id`.
- Партиционирование: нет.

Физически таблица хранит почти все поля исходного `articles.csv`, но уже в типизированном виде. Это фундамент для дальнейшего построения `silver.dim_article`.

### `bronze.hm_customers`

- Источник: `hive.raw.hm_customers_raw`.
- Grain: 1 строка = 1 клиентская запись.
- Назначение: технический справочник клиентов.
- Технические поля: `ingest_ts`, `source_file_name`, `batch_id`.
- Партиционирование: нет.

Слой нужен для построения `silver.dim_customer` и для контроля изменений по batch.

### `bronze.hm_transactions`

- Источник: `hive.raw.hm_transactions_raw`.
- Grain: 1 строка = 1 строка покупки.
- Бизнес-grain: `t_dat + customer_id + article_id + sales_channel_id`.
- Назначение: технический факт транзакций.
- Технические поля: `ingest_ts`, `source_file_name`, `batch_id`.
- Партиционирование: `month(t_dat)`.

Эта таблица уже является основой для всех дальнейших фактов и витрин по продажам.

## Silver

### Общая роль слоя

`silver` - это очищенная аналитическая модель. Здесь данные уже удобно использовать для соединений, расчётов и витрин.

На этом слое допустимы:

- дедупликация;
- стандартизация;
- производные признаки;
- построение фактов и измерений;
- календарные и агрегатные сущности.

### `silver.dim_article`

- Источник: `bronze.hm_articles`.
- Grain: 1 строка = 1 `article_id`.
- Назначение: основной товарный справочник для аналитики.
- Производные поля: `is_ladieswear`, `is_menswear`, `is_kids`, `color_family`.

Что делает запрос наполнения:

- берёт только строки, относящиеся к текущему batch;
- выбирает по каждому `article_id` самую свежую запись по `ingest_ts`, `batch_id`, `source_file_name`;
- вычисляет флаги товарных групп по регулярным выражениям;
- заполняет `color_family` как `NULL`.

Это не просто копия `bronze`, а уже аналитически пригодная размерность.

### `silver.dim_customer`

- Источник: `bronze.hm_customers`.
- Grain: 1 строка = 1 `customer_id`.
- Назначение: основной клиентский справочник для аналитики.
- Производные поля: `age_band`, `is_active_customer`, `is_fn_flag_present`.

Что делает запрос наполнения:

- выбирает по каждому `customer_id` наиболее свежую запись;
- группирует возраст в предопределённые диапазоны;
- переводит `active` в булевый признак активности;
- проверяет, присутствует ли значение флага `fn`.

### `silver.dim_date`

- Источник: календарь, построенный из дат транзакций в `bronze.hm_transactions`.
- Grain: 1 строка = 1 календарная дата.
- Назначение: календарное измерение для аналитики по времени.
- Поля: `date_day`, `date_year`, `date_month`, `date_day_of_month`, `date_day_of_week`, `week_of_year`.

Заполнение идёт не из внешнего календаря, а из фактического диапазона дат в транзакциях. Это делает таблицу зависимой от реальных данных, а не от искусственного диапазона.

### `silver.fact_sales_line`

- Источник: `bronze.hm_transactions`.
- Grain: 1 строка = 1 строка покупки.
- Ключевые поля: `sale_date`, `customer_id`, `article_id`, `sales_channel_id`.
- Назначение: основной факт продаж для дальнейших витрин.
- Партиционирование: `month(sale_date)`.

По смыслу это почти прямой перенос транзакций в аналитический слой, но уже в структуре, удобной для дальнейших расчётов.

Особенность обновления:

- факт пересобирается только по затронутым месячным частям;
- фильтр по месяцам передаётся из Python-лоадера как динамический предикат.

### `silver.fact_customer_article_stats`

- Источник: агрегат поверх `silver.fact_sales_line`.
- Grain: 1 строка = `customer_id + article_id`.
- Назначение: статистика повторных покупок по паре клиент - товар.
- Поля: `first_purchase_date`, `last_purchase_date`, `purchase_cnt`, `total_revenue`, `avg_price`.

Эта таблица важна, потому что она превращает поток транзакций в компактную поведенческую статистику.

Стратегия наполнения здесь двухуровневая:

- для нового batch используется `MERGE` поверх агрегата по текущему batch;
- для повторной пересборки уже существующего batch используется безопасный частичный пересчёт по префиксам `customer_id`.

Это одна из самых нетривиальных частей проекта, потому что здесь соединяются инкрементальная логика и защита от повторной обработки.

## Marts

### Общая роль слоя

`marts` - это верхний аналитический слой для BI и семантической публикации.

В проекте он строится только из `silver`, а не напрямую из `bronze`.

Общий принцип такой:

- одна витрина = один понятный бизнес-вопрос;
- grain должен быть явно фиксирован;
- метрики должны быть стабильными и читаемыми для BI-слоя.

### `mart.sales_daily_channel`

- Источник: `silver.fact_sales_line`.
- Grain: 1 строка = `sale_date + sales_channel_id`.
- Назначение: дневные продажи по каналам продаж.
- Метрики: `revenue`, `items_sold`, `buyers_cnt`, `avg_item_price`.
- Партиционирование: `month(sale_date)`.

Здесь `items_sold` означает количество строк покупок, а не количество физических единиц товара.

### `mart.sales_monthly_category`

- Источники: `silver.fact_sales_line`, `silver.dim_article`.
- Grain: 1 строка = `sale_month + category`.
- Назначение: месячная динамика продаж по товарным категориям.
- Метрики: `revenue`, `items_sold`, `buyers_cnt`, `active_sku_cnt`.
- Партиционирование: `month(sale_month)`.

Поле `category` в текущей реализации заполняется из `product_group_name`.

### `mart.customer_segment_monthly`

- Источники: `silver.fact_sales_line`, `silver.dim_customer`.
- Grain: 1 строка = `sale_month + customer_segment`.
- Назначение: месячная аналитика по клиентским сегментам.
- Метрики: `revenue`, `buyers_cnt`, `purchase_lines_cnt`, `revenue_per_buyer`, `avg_item_price`.
- Партиционирование: `month(sale_month)`.

Здесь важно ограничение текущей реализации: `customer_segment` строится как `COALESCE(club_member_status, 'unknown')`. Это не сложная поведенческая сегментация, а упрощённая категория на базе статуса участника клуба.

### `mart.repeat_purchase_category`

- Источники: `silver.fact_customer_article_stats`, `silver.dim_article`.
- Grain: 1 строка = 1 категория.
- Назначение: аналитика повторных покупок по товарным категориям.
- Метрики: `repeat_pairs_cnt`, `repeat_customers_cnt`, `avg_purchase_cnt`, `repeat_revenue`.
- Партиционирование: нет.

В витрину попадают только пары `customer_id + article_id`, у которых `purchase_cnt > 1`.

### `mart.customer_rfm_monthly`

- Источник: `silver.fact_sales_line`.
- Grain: 1 строка = `customer_id + snapshot_month`.
- Назначение: RFM-профиль клиента на конец месяца.
- Метрики и признаки: `recency_days`, `frequency_365d`, `monetary_365d`, `rfm_segment`.
- Партиционирование: `month(snapshot_month)`.

Особенность расчёта:

- снимок строится на последний день месяца;
- окно `frequency_365d` и `monetary_365d` охватывает последние 365 дней включительно;
- сегменты заданы вручную порогами в SQL.

## Что важно помнить о таблицах

В проекте нет физически закреплённых `PRIMARY KEY` и `FOREIGN KEY` в DDL. Это означает, что:

- grain и ключи определяются логически;
- целостность держится на SQL-логике наполнения;
- корректность модели нужно объяснять через правила загрузки и назначения таблиц, а не через ограничения СУБД.

Ещё несколько практических ограничений:

- `raw` не имеет отдельного постоянного DDL как таблица;
- `color_family` в `silver.dim_article` пока не вычисляется и остаётся `NULL`;
- `customer_segment` в `mart.customer_segment_monthly` является упрощённым признаком;
- `marts` физически лежат в схеме `mart`, хотя логически это единый верхний слой;
- часть обновлений построена инкрементально, а часть пересобирается целиком.

## Краткая карта наполняемости

- `articles.csv` -> `raw` -> `bronze.hm_articles` -> `silver.dim_article` -> `mart.sales_monthly_category`, `mart.repeat_purchase_category`
- `customers.csv` -> `raw` -> `bronze.hm_customers` -> `silver.dim_customer` -> `mart.customer_segment_monthly`
- `transactions_train.csv` -> `raw` -> `bronze.hm_transactions` -> `silver.fact_sales_line` -> `mart.sales_daily_channel`, `mart.sales_monthly_category`, `mart.customer_segment_monthly`, `mart.customer_rfm_monthly`
- `silver.fact_sales_line` -> `silver.fact_customer_article_stats` -> `mart.repeat_purchase_category`

Этот маршрут и есть основа всей аналитической модели проекта.
