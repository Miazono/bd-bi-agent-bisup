# Конфигурация и служебные Python-модули

## Зачем этот документ

Этот материал разбирает общую Python-основу проекта: конфигурацию, клиентские обертки и служебные скрипты, которые поддерживают весь конвейер загрузки данных.
Здесь важно понять не только отдельные функции, но и то, как Python связывает MinIO, Trino, SQL-шаблоны и подготовку хранилища.

Разбор опирается на фактические файлы:

- [`config/settings.py`](/root/repos/db-bi-agent-bisup/config/settings.py)
- [`ingestion/utils/s3_client.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/s3_client.py)
- [`ingestion/utils/trino_client.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/trino_client.py)
- [`ingestion/utils/sql_assets.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/sql_assets.py)
- [`ingestion/utils/sql_templates.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/sql_templates.py)
- [`scripts/init_storage.py`](/root/repos/db-bi-agent-bisup/scripts/init_storage.py)
- [`scripts/gen_schema.py`](/root/repos/db-bi-agent-bisup/scripts/gen_schema.py)

## Общая роль Python-основы

В проекте Python не является местом, где живёт основная бизнес-логика преобразований.
Его задача другая:

- загрузить и нормализовать настройки;
- собрать подключения к MinIO и Trino;
- найти SQL-артефакты внутри репозитория;
- подставить параметры в SQL-шаблоны;
- подготовить бакет и служебные префиксы;
- сгенерировать производную документацию по физическим таблицам.

Такое разделение делает конвейер читаемым:

- SQL отвечает за преобразование данных;
- Python отвечает за оркестрацию, доступ к сервисам и повторяемость запуска.

## `config/settings.py`

### Назначение

[`config/settings.py`](/root/repos/db-bi-agent-bisup/config/settings.py) — это единая точка, где проект собирает переменные окружения в структурированный объект `Settings`.
Этот модуль нужен, чтобы не читать `.env` и переменные окружения по всему коду вручную.

### Как загружается окружение

При импорте модуля выполняется `_load_environment()`.
Она:

- читает `APP_ENV`;
- приводит значение к нижнему регистру;
- разрешает только `local` и `docker`;
- выбирает файл `.env.local` или `.env.docker`;
- загружает его через `load_dotenv(..., override=False)`.

Это означает, что проект поддерживает два режима работы:

- локальный запуск Python-скриптов;
- запуск контейнеров внутри docker-сети.

### Что хранит `Settings`

Класс `Settings` содержит параметры, нужные почти всем частям пайплайна:

- имя бакета и префиксы слоёв;
- настройки MinIO и S3-совместимого доступа;
- URI Hive Metastore;
- параметры Trino;
- адрес WrenAI;
- ключ LLM-провайдера для BI-контура.

Структура намеренно централизована: если какой-то модуль зависит от хранилища или Trino, он берёт настройки отсюда.

### Что важно в реализации

В модуле есть два уровня значений:

- значения из окружения;
- значения по умолчанию.

Это удобно для локальной разработки, но важно помнить, что часть дефолтов — учебные и не равны реальным секретам или боевой конфигурации.

### Практический нюанс текущей реализации

В коде есть одна деталь, которую полезно заметить отдельно.

В [`config/settings.py`](/root/repos/db-bi-agent-bisup/config/settings.py) значение для HTTP-схемы Trino читается из переменной `TRINO_HTTP_SCHEMA`, а в примерах окружения используется имя `TRINO_HTTP_SCHEME`.

Это означает, что:

- в текущей реализации есть расхождение в названии переменной;
- на практике проект часто опирается на `TRINO_COORDINATOR`, поэтому проблема может быть незаметна;
- при ручной отладке лучше смотреть не только на `.env.example`, но и на то, как именно переменная читается в коде.

### Полезные свойства

В `Settings` есть свойства:

- `raw_base_uri`;
- `bronze_base_uri`;
- `silver_base_uri`;
- `mart_base_uri`.

Они собирают базовые `s3a://`-пути к слоям.
Это упрощает использование единого соглашения о расположении данных.

### Ограничения

Важное ограничение этой схемы:

- она ожидает, что нужные переменные действительно заданы;
- она не пытается угадать, какой сервис должен быть доступен;
- она не делает сложную валидацию логики конфигурации;
- она лишь собирает и типизирует уже подготовленные значения.

То есть `settings.py` — это конфигурационный слой, а не менеджер окружения общего назначения.

## `ingestion/utils/s3_client.py`

### Назначение

[`ingestion/utils/s3_client.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/s3_client.py) содержит обертку над MinIO-клиентом.
Её задача — не просто подключиться к объектному хранилищу, а дать удобные методы для типичных операций проекта.

### Главные элементы

В модуле есть:

- dataclass `S3ObjectInfo`;
- класс `S3Client`.

`S3ObjectInfo` хранит краткую информацию об объекте:

- ключ;
- размер;
- `etag`.

### Что делает `S3Client`

Класс собирает MinIO-клиент из параметров окружения и предоставляет методы для типовых операций:

- проверка существования бакета;
- создание бакета;
- проверка объекта;
- чтение метаданных объекта;
- список объектов по префиксу;
- загрузка файла;
- сборка `s3a://` URI для таблиц и каталогов.

### Почему это полезно

Без этой обертки каждый загрузочный скрипт вынужден был бы повторять одинаковую логику:

- как преобразовать endpoint;
- как нормализовать ключи;
- как проверить бакет;
- как построить путь к объекту;
- как собрать URI для Trino.

Обертка делает этот слой единообразным и уменьшает риск расхождения в путях и адресах.

### Важные детали реализации

Несколько моментов стоит понимать отдельно:

- `normalize_key()` убирает ведущий `/`, чтобы ключи были в едином формате.
- `bucket_exists()` и `object_exists()` защищены от `S3Error`.
- `head_object()` возвращает краткую структуру с размером и `etag`.
- `build_table_uri()` и `build_dir_uri()` помогают формировать адреса, которые понимает Trino и Iceberg.

### Ограничения

У этой обертки нет полноценного слоя доменной валидации.
Она не знает, что именно должно лежать в бакете по смыслу, и не контролирует бизнес-структуру данных.
Её задача техническая: безопасно и удобно общаться с MinIO.

## `ingestion/utils/trino_client.py`

### Назначение

[`ingestion/utils/trino_client.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/trino_client.py) содержит обертку над `trino.dbapi`.
Она используется во всех шагах, где Python должен выполнять SQL через Trino.

### Что делает клиент

`TrinoClient`:

- читает параметры подключения из `Settings`;
- поддерживает подключение через `TRINO_COORDINATOR`;
- при необходимости может собирать адрес из `TRINO_HOST`, `TRINO_PORT`, `TRINO_HTTP_SCHEMA`;
- выполняет SQL-строки;
- выполняет SQL-файлы;
- возвращает результаты через `fetchone()` и `fetchall()`.

### Почему это выделено в отдельный класс

Такой подход позволяет:

- не повторять логику подключения в каждом скрипте;
- единообразно обрабатывать выполнение SQL;
- скрыть техническую разницу между строкой SQL и SQL-файлом;
- держать код пайплайна коротким и читаемым.

### Что важно в `execute()`

Метод `execute()`:

- обрезает лишние пробелы;
- убирает завершающую `;`;
- выполняет SQL через cursor;
- пытается прочитать результат через `fetchall()`;
- если результат не нужен или не возвращается, отдаёт пустой список.

Это удобная унификация для `SELECT`, `INSERT`, `DELETE` и `MERGE`.

### Ограничения

Обертка не пытается быть ORM и не скрывает SQL.
Она просто даёт удобный способ выполнять запросы и получать результат.
Если нужен сложный контроль транзакций или особое поведение драйвера, это всё равно надо решать на уровне SQL и оркестрации.

## `ingestion/utils/sql_assets.py`

### Назначение

[`ingestion/utils/sql_assets.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/sql_assets.py) отвечает за поиск SQL-артефактов внутри репозитория.
Это маленький, но важный слой, который связывает Python с `sql/ddl/` и `sql/queries/`.

### Что делает класс `SqlAssets`

Класс:

- хранит корень репозитория;
- собирает путь из набора сегментов;
- передаёт найденный путь в шаблонизатор SQL.

### Зачем это нужно

Без этой утилиты Python-код был бы перегружен ручным склеиванием путей.
С ней же скрипты могут работать от корня репозитория, а не от текущей рабочей директории.

Это особенно полезно, потому что:

- DDL лежат отдельно от оркестрации;
- SQL-запросы лежат отдельно от Python;
- одна и та же логика нужна в нескольких загрузчиках.

## `ingestion/utils/sql_templates.py`

### Назначение

[`ingestion/utils/sql_templates.py`](/root/repos/db-bi-agent-bisup/ingestion/utils/sql_templates.py) реализует простую подстановку плейсхолдеров в SQL-файлы.

### Как это работает

Функция `render_sql_template()`:

- проверяет, что файл существует;
- читает SQL как текст;
- заменяет токены вида `__NAME__` на переданные значения;
- ищет необработанные токены;
- если что-то осталось, выбрасывает ошибку.

### Почему это удобно

Такой подход даёт:

- простую параметризацию SQL без тяжёлого шаблонизатора;
- явные и читаемые плейсхолдеры;
- контроль того, что никакой токен не забыт.

### Почему это важно именно для проекта

В этом репозитории SQL часто содержит операционные параметры:

- `batch_id`;
- имя файла-источника;
- длину префикса;
- дату или месячный фильтр.

Простая текстовая подстановка здесь лучше, чем сложный шаблонизатор, потому что запросы остаются читаемыми и почти не теряют SQL-вид.

### Ограничения

Этот механизм намеренно простой.
Он не умеет:

- условную логику;
- циклы;
- выражения;
- экранирование как полноценный шаблонизатор.

Зато он хорошо подходит для небольших и явно контролируемых SQL-подстановок в пайплайне.

## `scripts/init_storage.py`

### Назначение

[`scripts/init_storage.py`](/root/repos/db-bi-agent-bisup/scripts/init_storage.py) подготавливает структуру бакета в MinIO.
Это вспомогательный служебный скрипт, который нужен до загрузки данных.

### Что делает скрипт

Он:

- проверяет наличие бакета;
- создаёт бакет, если его нет;
- создаёт маркерные объекты `.keep` для префиксов `raw/`, `bronze/`, `silver/`, `marts/`;
- пишет понятные логи о выполненных действиях.

### Почему используются `.keep`

В объектном хранилище нет папок в привычном файловом смысле.
Маркер `.keep` помогает показать структуру префиксов и сохранить их как служебные точки организации слоёв.

### Почему это нужно

Этот скрипт делает старт хранилища воспроизводимым:

- сначала создаётся бакет;
- потом появляются префиксы слоёв;
- после этого уже можно запускать загрузчики raw, bronze, silver и marts.

### Ограничения

Скрипт не создаёт таблицы и не загружает данные.
Он отвечает только за физическую подготовку хранилища.

## `scripts/gen_schema.py`

### Назначение

[`scripts/gen_schema.py`](/root/repos/db-bi-agent-bisup/scripts/gen_schema.py) генерирует производный каталог физических таблиц по DDL-файлам.
Он не обращается к Trino и не требует поднятого стека.

### Что делает скрипт

Скрипт:

- читает `sql/ddl/**/*.sql`;
- извлекает схему, имя таблицы, список столбцов и свойства `WITH`;
- формирует текстовый каталог в `docs/data/catalog_generated.md`.

### Как устроен разбор DDL

Логика опирается на регулярные выражения и вспомогательные dataclass-структуры:

- `Column`;
- `TableDefinition`.

Скрипт парсит:

- секцию `CREATE TABLE IF NOT EXISTS`;
- блок столбцов;
- блок `WITH (...)`.

### Почему это полезно

Каталог `docs/data/catalog_generated.md` остаётся производным артефактом, который можно обновить автоматически.
Это снижает риск расхождений между DDL и документацией по физическим таблицам.

### Ограничения

Этот скрипт понимает только ожидаемый формат DDL.
Если структура файла станет сильно отличаться от текущего шаблона, парсер нужно будет обновлять вместе с DDL.

## Как все модули связаны с пайплайном

Если собрать картину целиком, получается такая цепочка:

1. `settings.py` загружает окружение и даёт общие параметры.
2. `S3Client` и `TrinoClient` дают доступ к MinIO и Trino.
3. `SqlAssets` и `sql_templates.py` позволяют находить и параметризовать SQL-файлы.
4. `init_storage.py` готовит бакет и префиксы.
5. Загрузочные скрипты используют всё это для построения `raw`, `bronze`, `silver` и `marts`.
6. `gen_schema.py` поддерживает актуальность документации по физической модели.

То есть эта Python-основа не конкурирует с SQL-логикой, а обслуживает её.

## Что важно запомнить

- `settings.py` — это единая конфигурация, а не просто набор глобальных переменных.
- `S3Client` и `TrinoClient` — это технические обертки над внешними сервисами.
- `SqlAssets` и `sql_templates.py` связывают Python с SQL-файлами репозитория.
- `init_storage.py` подготавливает объектное хранилище.
- `gen_schema.py` поддерживает производную документацию по DDL.

Именно этот слой делает пайплайн повторяемым, управляемым и удобным для объяснения на защите.
