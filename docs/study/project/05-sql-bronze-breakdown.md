# Разбор SQL для слоя bronze

## Зачем нужен этот документ

Этот материал разбирает именно ту часть проекта, которая переводит исходные CSV-файлы в слой `bronze`.
Здесь важны не только сами SQL-запросы, но и их место в конвейере: от временных внешних таблиц `hive.raw.*` до физических таблиц `iceberg.bronze.*`.

Разбор опирается только на фактическую реализацию репозитория:

- `sql/queries/bronze/load_hm_articles.sql`;
- `sql/queries/bronze/load_hm_customers.sql`;
- `sql/queries/bronze/load_hm_transactions.sql`;
- `sql/ddl/bronze/*.sql`;
- `ingestion/load_bronze.py`.

## Роль bronze-запросов

SQL-запросы слоя `bronze` выполняют техническую нормализацию исходных данных.
Их задача не в том, чтобы делать бизнес-аналитику, а в том, чтобы:

- прочитать CSV через SQL;
- привести типы к нужным значениям;
- заменить пустые строки на `NULL`;
- добавить технические поля загрузки;
- записать результат в физические таблицы Iceberg;
- оставить данные близкими к источнику, чтобы следующий слой мог строиться уже на чистой основе.

Именно поэтому `bronze` занимает промежуточное положение между сырыми файлами и аналитической моделью.

## Как данные попадают в bronze

Поток устроен в два шага.

Сначала `ingestion/load_bronze.py` создаёт временные внешние таблицы `hive.raw.*`, которые читают CSV-файлы из MinIO.
Потом уже SQL-запросы из `sql/queries/bronze/` переносят данные из этих временных таблиц в `iceberg.bronze.*`.

Итоговая схема выглядит так:

`raw` в MinIO -> `hive.raw.hm_*_raw` -> `iceberg.bronze.hm_*`

Важный момент: `hive.raw.*` не являются самостоятельным аналитическим слоем.
Это технический мост между файловым хранением и физическими таблицами.

## Как создаются временные внешние таблицы

Логика создания внешних таблиц находится в [`ingestion/load_bronze.py`](/root/repos/db-bi-agent-bisup/ingestion/load_bronze.py).
Скрипт:

- проверяет наличие raw-файлов в MinIO;
- создаёт схему `hive.raw`;
- создаёт схему `iceberg.bronze`;
- удаляет и заново создаёт внешние таблицы `hive.raw.hm_articles_raw`, `hive.raw.hm_customers_raw`, `hive.raw.hm_transactions_raw`;
- указывает им CSV-формат, путь к файлам и число пропускаемых строк заголовка.

Структура внешних таблиц жёстко совпадает с колонками исходных CSV.
Это важно, потому что на этом этапе SQL читает файл почти как плоский текстовый источник, а не как уже нормализованную модель.

## DDL слоя bronze

DDL физических таблиц находится в `sql/ddl/bronze/`.
Все три таблицы созданы как `CREATE TABLE IF NOT EXISTS ... WITH (format = 'PARQUET', location = ...)`.

### `bronze.hm_articles`

- хранит товарный справочник;
- содержит почти все поля `articles.csv`;
- включает технические поля `ingest_ts`, `source_file_name`, `batch_id`;
- не имеет партиционирования.

### `bronze.hm_customers`

- хранит клиентский справочник;
- содержит поля `customers.csv`;
- включает те же технические поля загрузки;
- не имеет партиционирования.

### `bronze.hm_transactions`

- хранит строки покупок;
- содержит дату транзакции, клиента, товар, цену и канал продаж;
- включает технические поля загрузки;
- партиционируется по `month(t_dat)`.

## `load_hm_articles.sql`

Файл [`sql/queries/bronze/load_hm_articles.sql`](/root/repos/db-bi-agent-bisup/sql/queries/bronze/load_hm_articles.sql) загружает данные из `hive.raw.hm_articles_raw` в `iceberg.bronze.hm_articles`.

### Что делает запрос

Запрос:

- явно перечисляет все целевые столбцы;
- приводит числовые поля к `BIGINT` или `INTEGER`;
- оставляет текстовые поля строками;
- заменяет пустые строки на `NULL`;
- добавляет `CURRENT_TIMESTAMP` в `ingest_ts`;
- подставляет `source_file_name` и `batch_id` через шаблонные плейсхолдеры.

### Какие типы приводятся

В этом запросе преобразуются, например:

- `article_id` -> `BIGINT`;
- `product_code` -> `BIGINT`;
- `product_type_no` -> `INTEGER`;
- `graphical_appearance_no` -> `INTEGER`;
- `colour_group_code` -> `INTEGER`;
- `perceived_colour_value_id` -> `INTEGER`;
- `perceived_colour_master_id` -> `INTEGER`;
- `department_no` -> `INTEGER`;
- `index_group_no` -> `INTEGER`;
- `section_no` -> `INTEGER`;
- `garment_group_no` -> `INTEGER`.

### Где нормализуются пустые значения

Почти все поля проходят через `NULLIF(col, '')`.
Это означает, что пустая строка из CSV становится настоящим `NULL`, а не остаётся пустым текстом.

Такой приём особенно важен для числовых столбцов, потому что сначала убирается пустое значение, а затем выполняется `CAST`.
Без этого преобразования пустая строка могла бы сломать чтение или превратиться в некорректное значение.

### Какие техполя добавляются

В конце запроса добавляются:

- `ingest_ts`;
- `source_file_name`;
- `batch_id`.

Это даёт возможность понимать, когда и из какого файла пришла строка, а также к какому батчу она относится.

### Что важно понимать об этом запросе

Этот запрос не делает бизнес-классификацию и не пытается определить дополнительные признаки.
Он отвечает только за техническую подготовку справочника товаров.

## `load_hm_customers.sql`

Файл [`sql/queries/bronze/load_hm_customers.sql`](/root/repos/db-bi-agent-bisup/sql/queries/bronze/load_hm_customers.sql) загружает данные из `hive.raw.hm_customers_raw` в `iceberg.bronze.hm_customers`.

### Что делает запрос

Запрос:

- читает исходные поля клиентов;
- чистит пустые строки;
- убирает хвост `.0` из некоторых полей;
- приводит значения к числу там, где это возможно;
- добавляет технические поля загрузки.

### Какие типы приводятся

Здесь преобразуются:

- `fn` -> `INTEGER`;
- `active` -> `INTEGER`;
- `age` -> `INTEGER`.

### Где нормализуются пустые значения

Пустые значения сначала проходят через `NULLIF(trim(...), '')`.
Это позволяет корректно обработать как пустую строку, так и строку из пробелов.

После этого для некоторых полей используется `regexp_replace(..., '\\.0+$', '')`, чтобы убрать типичный хвост `.0`.
Затем результат приводится через `TRY_CAST`.

Это полезно для полей, которые в CSV могут выглядеть как числовые значения в текстовом виде, хотя фактически должны храниться как целые числа.

### Почему используется `TRY_CAST`

`TRY_CAST` безопаснее обычного `CAST`, потому что при неудачном преобразовании он не падает с ошибкой, а возвращает `NULL`.
Это помогает сделать загрузку более устойчивой к грязным или неожиданным значениям в исходных данных.

### Какие техполя добавляются

Как и в других бронзовых запросах, добавляются:

- `ingest_ts`;
- `source_file_name`;
- `batch_id`.

### Что важно понимать об этом запросе

Этот запрос уже немного сложнее, чем загрузка статического справочника товаров, потому что здесь данные из исходного файла требуют более аккуратной очистки.
Но и здесь слой остаётся техническим: он не строит поведенческие признаки, а только подготавливает данные клиента к следующему уровню модели.

## `load_hm_transactions.sql`

Файл [`sql/queries/bronze/load_hm_transactions.sql`](/root/repos/db-bi-agent-bisup/sql/queries/bronze/load_hm_transactions.sql) загружает транзакции из `hive.raw.hm_transactions_raw` в `iceberg.bronze.hm_transactions`.

### Что делает запрос

Запрос:

- читает строки покупок из внешней таблицы;
- приводит дату к типу `DATE`;
- приводит цену к `DECIMAL(12,4)`;
- приводит идентификатор товара и канал продаж к числу;
- заменяет пустые строки на `NULL`;
- добавляет технические поля загрузки.

### Какие типы приводятся

Здесь преобразуются:

- `t_dat` -> `DATE`;
- `article_id` -> `BIGINT`;
- `price` -> `DECIMAL(12,4)`;
- `sales_channel_id` -> `INTEGER`.

### Где нормализуются пустые значения

Почти все поля проходят через `NULLIF(col, '')`.
Это нужно, чтобы пустой CSV-элемент не ломал приведение типов.

### Какие техполя добавляются

В конец строки, как и в других bronze-загрузках, добавляются:

- `ingest_ts`;
- `source_file_name`;
- `batch_id`.

### Особенность этой таблицы

Именно `bronze.hm_transactions` становится основой для аналитических фактов и витрин по продажам.
Поэтому здесь особенно важно, что дата и цена приводятся в корректные типы, а данные могут быть физически разложены по месячным партициям.

## Общие правила для bronze-загрузок

У всех трёх SQL-запросов одинаковая архитектурная роль:

- работать рядом с источником;
- не добавлять сложную бизнес-логику;
- сохранять структуру данных, близкую к CSV;
- подготавливать строку к следующему аналитическому уровню.

Общее поведение тоже одинаковое:

- пустые строки нормализуются в `NULL`;
- типы приводятся явно;
- загружается техническая метаинформация;
- запрос строится как `INSERT INTO ... SELECT ...`.

## Ограничения слоя bronze

У `bronze` есть сознательные ограничения.

- Здесь нет полноценной бизнес-агрегации.
- Здесь нет сложного моделирования измерений и фактов.
- Здесь не строится семантический слой.
- Здесь не решаются вопросы удобства BI-агента.
- Здесь нет физически закреплённых `PRIMARY KEY` и `FOREIGN KEY`.

Это нормальная часть архитектуры: `bronze` должен оставаться техническим слоем, а не превращаться в гибрид всего подряд.

## Что особенно важно запомнить

Для защиты проекта важно уметь объяснить три вещи:

- `bronze` не равен raw-файлу, потому что данные уже типизированы и загружены в таблицу;
- `bronze` не равен `silver`, потому что здесь нет бизнес-очистки и производных признаков;
- `hive.raw.*` не являются постоянными таблицами проекта, а существуют как технический мост для чтения CSV.

Именно это и делает бронзовый слой полезным: он отделяет техническую загрузку от аналитической модели.
