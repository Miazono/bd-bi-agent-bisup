# Разбор `load_silver.py`

## Зачем этот документ

Этот материал разбирает Python-скрипт [`ingestion/load_silver.py`](/root/repos/db-bi-agent-bisup/ingestion/load_silver.py) как оркестратор слоя `silver`.
Здесь важно не только понять, какие функции есть в коде, но и увидеть, как они связывают `bronze`, SQL-файлы и физические таблицы `silver`.

Разбор опирается только на фактическую реализацию репозитория:

- [`ingestion/load_silver.py`](/root/repos/db-bi-agent-bisup/ingestion/load_silver.py)
- [`sql/ddl/silver/`](/root/repos/db-bi-agent-bisup/sql/ddl/silver)
- [`sql/queries/silver/`](/root/repos/db-bi-agent-bisup/sql/queries/silver)

## Роль скрипта в конвейере

`load_silver.py` — это не просто загрузчик данных, а оркестратор аналитического слоя.
Его задача состоит в том, чтобы:

- проверить, что нужный batch уже есть в `bronze`;
- создать схему `iceberg.silver`, если она отсутствует;
- применить DDL для таблиц `silver`;
- обновить размерности;
- пересобрать факт продаж по месячным частям;
- обновить производный агрегат по клиенту и товару;
- логировать результат выполнения.

Именно на этом шаге данные из технического слоя начинают превращаться в аналитическую модель.

## Как устроен файл

Скрипт удобно читать как последовательность четырёх частей:

1. настройки и общие вспомогательные функции;
2. функции для подготовки схемы и таблиц;
3. функции обновления каждой таблицы `silver`;
4. аргументы запуска и `main()`.

Такое разбиение делает файл длинным, но логически прозрачным: в нём видно и оркестрацию, и связь с SQL.

## Настройки и общие объекты

В начале файла:

- настраивается логирование через `logging.basicConfig(...)`;
- берутся значения из [`config/settings.py`](/root/repos/db-bi-agent-bisup/config/settings.py);
- создаётся экземпляр `SqlAssets`, чтобы удобно находить SQL-файлы по путям в репозитории.

Ключевые константы:

- `LAKEHOUSE_BUCKET` — имя бакета;
- `SILVER_PREFIX` — префикс слоя `silver`;
- `SQL_ASSETS` — помощник для поиска и рендера SQL.

Это означает, что скрипт не хранит пути к SQL “вручную” и не завязан на случайные относительные директории.

## Вспомогательные функции

### `q(value)`

Функция `q()` экранирует одинарные кавычки в строке.
Она нужна для безопасной подстановки значений в SQL-шаблоны и запросы, где строка вставляется прямо в текст.

### `execute_step(trino, sql, title)`

Функция выполняет один SQL-шаг и пишет в лог, что шаг начался и завершился.
Это делает оркестрацию читаемой, потому что в логах видно не просто “выполнился SQL”, а именно конкретный этап пайплайна.

### `log_row_count(trino, table_name)`

Считает количество строк в целевой таблице и пишет число в лог.
Это простая, но полезная проверка: после обновления размерности или календаря можно быстро увидеть, что таблица не пустая.

### `parse_fqn(table_name)`

Разбирает fully qualified name вида `catalog.schema.table`.
Функция нужна для работы с Iceberg metadata tables, например с `"$files"` у таблиц `silver` и `mart`.

### `log_iceberg_files_summary(trino, table_name)`

Смотрит системную таблицу Iceberg `"$files"` и логирует:

- суммарное число строк в файлах;
- количество файлов;
- общий размер.

Это не бизнес-проверка, а техническая валидация того, что таблица действительно материализована в Iceberg.

### `log_batch_row_count(trino, table_name, batch_id)`

Считает строки только для указанного `batch_id`.
Это полезно именно для `fact_sales_line`, потому что таблица обновляется по batch и месяцам.

## Проверка batch

### `validate_bronze_batch(trino, batch_id)`

Перед построением `silver` скрипт проверяет, что batch действительно существует в `bronze`.
Проверяются три таблицы:

- `iceberg.bronze.hm_articles`;
- `iceberg.bronze.hm_customers`;
- `iceberg.bronze.hm_transactions`.

Для каждой таблицы считается количество строк с нужным `batch_id`.
Если хотя бы в одной таблице строк нет, скрипт завершится ошибкой.

Это важный инвариант:

- `silver` не должен строиться из неполного batch;
- логика построения предполагает, что batch последователен во всех источниках;
- ошибка на этом шаге защищает от частично загруженных данных.

## Создание схемы и таблиц

### `ensure_silver_schema(trino)`

Функция создаёт схему `iceberg.silver`, если её ещё нет.
В `WITH (location = ...)` указывается путь в MinIO, где будут храниться файлы таблиц `silver`.

Для проекта это означает:

- схема может быть создана повторно без повреждения данных;
- физическое расположение `silver` фиксировано и согласовано с `config/settings.py`;
- слой `silver` хранится отдельно от `bronze` и `marts`.

### `silver_ddl_paths()`

Эта функция возвращает список DDL-файлов:

- `silver_dim_article.sql`;
- `silver_dim_customer.sql`;
- `silver_dim_date.sql`;
- `silver_fact_sales_line.sql`;
- `silver_fact_customer_article_stats.sql`.

Это фактический перечень таблиц слоя `silver`.

### `ensure_silver_tables(trino)`

Функция проходит по списку DDL и применяет каждый файл через `trino.execute_file(...)`.
Если какой-то DDL-файл отсутствует, скрипт падает с ошибкой.

Это полезно, потому что структура таблиц не “угадывается” на ходу, а всегда создаётся из явных файлов в репозитории.

## Обновление `dim_article`

### Связанные SQL-файлы

- [`sql/queries/silver/refresh_dim_article_delete.sql`](/root/repos/db-bi-agent-bisup/sql/queries/silver/refresh_dim_article_delete.sql)
- [`sql/queries/silver/refresh_dim_article_insert.sql`](/root/repos/db-bi-agent-bisup/sql/queries/silver/refresh_dim_article_insert.sql)

### `refresh_dim_article(trino, batch_id)`

Обновление размерности товара сделано в два шага:

1. удалить строки по затронутым `article_id`;
2. вставить заново актуальные строки.

Такой подход проще читать и ревьюить, чем сложный частичный `UPDATE`.

### Что делает удаление

Удаляются все строки из `silver.dim_article`, у которых `article_id` встречается в текущем batch `bronze.hm_articles`.
Это подготавливает таблицу к вставке новой актуальной версии.

### Что делает вставка

Вставка строится вокруг двух идей:

- сначала выбираются только затронутые `article_id`;
- потом через `row_number() over (...)` выбирается самая свежая строка на каждый ключ.

Сортировка идёт по:

- `ingest_ts DESC`;
- `batch_id DESC`;
- `source_file_name DESC`.

После этого вычисляются флаги:

- `is_ladieswear`;
- `is_menswear`;
- `is_kids`.

Поле `color_family` в текущей реализации остаётся `NULL`.

### Почему это сделано так

Для размерности товара важна актуальность записи, а не просто наличие всех версий.
Поэтому логика “удалить и вставить заново” хорошо подходит для повторяемого batch-процесса.

## Обновление `dim_customer`

### Связанные SQL-файлы

- [`sql/queries/silver/refresh_dim_customer_delete.sql`](/root/repos/db-bi-agent-bisup/sql/queries/silver/refresh_dim_customer_delete.sql)
- [`sql/queries/silver/refresh_dim_customer_insert.sql`](/root/repos/db-bi-agent-bisup/sql/queries/silver/refresh_dim_customer_insert.sql)

### `refresh_dim_customer(trino, batch_id)`

Для клиентов используется тот же паттерн, что и для товара:

- удалить затронутые `customer_id`;
- вставить заново актуальные строки.

### Что делает вставка

Запрос выбирает самые свежие записи по `customer_id`, затем строит производные поля:

- `age_band`;
- `is_active_customer`;
- `is_fn_flag_present`.

Здесь особенно хорошо видно, что бизнес-логика уже живёт в SQL, а не в Python.

## Обновление `dim_date`

### Связанный SQL-файл

- [`sql/queries/silver/upsert_dim_date.sql`](/root/repos/db-bi-agent-bisup/sql/queries/silver/upsert_dim_date.sql)

### `upsert_dim_date(trino, batch_id)`

Календарная размерность пополняется через отдельный SQL-шаг.
Она не пересобирается целиком, а только добавляет отсутствующие даты.

### Как это работает

Запрос берёт минимальную и максимальную даты в `bronze.hm_transactions` для текущего batch, строит последовательность дат и вставляет только те дни, которых ещё нет в `silver.dim_date`.

### Почему это важно

Календарь должен быть стабильным и расширяться по мере появления новых данных, а не каждый раз пересоздаваться заново.

## Пересборка `fact_sales_line`

### Связанные SQL-файлы

- [`sql/queries/silver/refresh_fact_sales_line_delete.sql`](/root/repos/db-bi-agent-bisup/sql/queries/silver/refresh_fact_sales_line_delete.sql)
- [`sql/queries/silver/refresh_fact_sales_line_insert.sql`](/root/repos/db-bi-agent-bisup/sql/queries/silver/refresh_fact_sales_line_insert.sql)

### Общая идея

`fact_sales_line` обновляется не целиком, а по месячным частям.
Это уменьшает объём работы и делает повторные прогоны управляемыми.

### `get_batch_months(trino, batch_id)`

Функция находит все месяцы, присутствующие в `bronze.hm_transactions` для текущего batch.
Месяц вычисляется как `date_trunc('month', t_dat)`.

### `parse_months_arg(months_arg)`

Скрипт умеет принимать параметр `--months`.
Он ожидает список дат начала месяцев в формате `YYYY-MM-DD`, например:

```text
2020-08-01,2020-09-01
```

### `resolve_months_to_process(batch_months, requested_months)`

Если месяцы не переданы, скрипт берёт все месяцы batch.
Если месяцы переданы, он проверяет, что они действительно есть в batch, и сохраняет их в порядке batch.

### `month_filter(column_name, month_start)`

Эта функция строит предикат для одного месяца:

- дата должна быть не меньше начала месяца;
- дата должна быть меньше начала следующего месяца.

Это безопасный способ отфильтровать ровно нужный месяц.

### `refresh_fact_sales_line_month(...)`

Для каждого месяца выполняются два шага:

1. удалить строки из `silver.fact_sales_line` за этот месяц и batch;
2. вставить строки из `bronze.hm_transactions` за тот же месяц и batch.

### Что делает вставка

Запрос просто переносит транзакции из `bronze` в `silver`, переименовывая `t_dat` в `sale_date`.
Это почти прямой факт продаж, но уже в аналитическом слое.

### Почему это сделано по месяцам

Это компромисс между простотой и производительностью:

- полная пересборка всего факта была бы тяжелее;
- месячный фильтр прозрачен и удобно контролируется;
- логика хорошо согласуется с физическим партиционированием таблицы.

## Обновление `fact_customer_article_stats`

### Связанные SQL-файлы

- [`sql/queries/silver/merge_fact_customer_article_stats_batch_delta.sql`](/root/repos/db-bi-agent-bisup/sql/queries/silver/merge_fact_customer_article_stats_batch_delta.sql)
- [`sql/queries/silver/delete_impacted_stats_prefix.sql`](/root/repos/db-bi-agent-bisup/sql/queries/silver/delete_impacted_stats_prefix.sql)
- [`sql/queries/silver/insert_impacted_stats_prefix.sql`](/root/repos/db-bi-agent-bisup/sql/queries/silver/insert_impacted_stats_prefix.sql)

### Почему тут два сценария

Для этой таблицы используются две стратегии обновления:

- `MERGE` для нового batch;
- безопасная пересборка по префиксам `customer_id` для повторного batch.

Это самая нетривиальная часть скрипта.

### `fact_sales_line_batch_exists(trino, batch_id)`

Скрипт проверяет, есть ли уже строки данного batch в `silver.fact_sales_line`.
Если batch уже загружался, значит для агрегата нужен безопасный путь пересборки.

### `merge_fact_customer_article_stats_batch_delta(trino, batch_id)`

Если batch новый, агрегат обновляется через `MERGE`.
SQL считает по текущему batch:

- первую дату покупки;
- последнюю дату покупки;
- количество покупок;
- суммарную выручку;
- среднюю цену.

Если ключ уже существует, значения обновляются с учётом старого и нового состояния.
Если ключа нет, строка вставляется.

### `get_impacted_prefixes(trino, batch_id, prefix_len)`

Для повторного batch скрипт собирает префиксы `customer_id`, которые были затронуты в `bronze`.
Это ограничивает область пересборки.

### `refresh_fact_customer_article_stats_incremental(...)`

Если batch уже был загружен, скрипт идёт по затронутым префиксам и:

- удаляет соответствующие пары `customer_id + article_id`;
- пересчитывает их заново из `silver.fact_sales_line`.

### Почему это важно

Такой сценарий защищает от неконтролируемого накопления дубликатов при повторных прогонах.
Префикс в этом месте — это технический фильтр области пересчёта, а не бизнес-сегментация.

## Аргументы запуска

### `--batch-id`

Обязательный аргумент.
Именно по нему скрипт понимает, какой batch нужно загрузить в `silver`.

### `--stats-prefix-len`

Длина префикса `customer_id`, используемого для безопасной пересборки агрегата.
По умолчанию это `2`.

### `--months`

Необязательный аргумент.
Позволяет пересобрать только отдельные месяцы `fact_sales_line`.

### `--skip-stats`

Позволяет пропустить обновление `fact_customer_article_stats`.
Это полезно для более быстрых проверок и smoke-запусков.

## Как устроен `main()`

Последовательность в `main()` очень показательная:

1. разобрать аргументы;
2. создать `TrinoClient`;
3. проверить наличие batch в `bronze`;
4. создать схему `iceberg.silver`;
5. создать таблицы `silver`;
6. обновить `dim_article`;
7. обновить `dim_customer`;
8. обновить `dim_date`;
9. пересобрать `fact_sales_line` по месяцам;
10. при необходимости обновить `fact_customer_article_stats`;
11. залогировать итоги.

Это делает скрипт удобным для понимания: порядок шагов отражает иерархию самой модели данных.

## Ограничения реализации

У текущего решения есть несколько сознательных ограничений.

- Скрипт зависит от того, что batch уже полностью есть в `bronze`.
- `dim_article` и `dim_customer` обновляются через удаление и вставку, а не через более сложный инкрементальный `UPDATE`.
- `fact_sales_line` пересобирается по месяцам, а не по отдельным строкам или сложным диффам.
- `fact_customer_article_stats` использует префикс `customer_id` как технический механизм безопасной пересборки.
- `color_family` в `dim_article` пока не вычисляется и остаётся `NULL`.

Это не ошибки, а границы текущей версии реализации.

## Что важно запомнить

Если совсем коротко, `load_silver.py` делает три вещи:

- подтверждает, что batch из `bronze` полон и готов;
- вызывает SQL-логику построения размерностей, факта и агрегата;
- следит за тем, чтобы повторные прогоны не ломали модель.

Именно поэтому этот файл стоит рассматривать не как “ещё один скрипт загрузки”, а как центральную точку управления слоем `silver`.
