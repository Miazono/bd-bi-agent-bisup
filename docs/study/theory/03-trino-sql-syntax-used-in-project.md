# Синтаксис Trino SQL, используемый в проекте

## Назначение

Этот документ объясняет не весь SQL вообще, а только те конструкции Trino SQL, которые реально используются в проекте.
Его цель — помочь уверенно читать запросы из `sql/queries/` и понимать, что именно делает каждая группа операторов.

## Общий подход

В проекте SQL используется в трёх режимах:

- загрузка данных в таблицы через `INSERT INTO ... SELECT ...`;
- выборочное удаление строк через `DELETE`;
- обновление агрегатов через `MERGE`.

Почти вся фактическая бизнес-логика пайплайна выражена именно этими средствами.

## Базовые конструкции

### `SELECT`

`SELECT` выбирает и вычисляет набор столбцов.
В Trino это базовая конструкция для чтения, преобразования и агрегации данных.

Простейшая форма:

```sql
SELECT customer_id, article_id
FROM some_table
```

### `INSERT INTO ... SELECT ...`

Это основная конструкция загрузки в проекте.
Она означает: взять результат запроса `SELECT` и записать его в целевую таблицу.

Общий вид:

```sql
INSERT INTO target_table
SELECT ...
FROM source_table
```

Так в проекте загружаются:

- таблицы `bronze`;
- часть таблиц `silver`;
- все витрины `mart`.

### `DELETE`

`DELETE` удаляет строки по условию.
В проекте он часто используется перед повторной вставкой, чтобы безопасно пересобрать только затронутую часть данных.

Общий вид:

```sql
DELETE FROM target_table
WHERE some_condition
```

### `MERGE`

`MERGE` объединяет логику вставки и обновления.
Он нужен в ситуациях, когда для части ключей строка уже есть, а для части ещё нет.

Общий вид:

```sql
MERGE INTO target AS t
USING source AS s
ON ...
WHEN MATCHED THEN UPDATE SET ...
WHEN NOT MATCHED THEN INSERT (...)
VALUES (...)
```

В проекте `MERGE` особенно важен для обновления `silver.fact_customer_article_stats`.

## Приведение типов и очистка значений

### `CAST`

`CAST` жёстко приводит выражение к нужному типу.

Примеры:

```sql
CAST(NULLIF(article_id, '') AS BIGINT)
CAST(NULLIF(price, '') AS DECIMAL(12,4))
CAST(date_trunc('month', sale_date) AS DATE)
```

Если значение не удаётся привести, запрос может завершиться ошибкой.

### `TRY_CAST`

`TRY_CAST` тоже пытается привести тип, но при неудаче возвращает `NULL`, а не ошибку.
Это удобно для сырых данных, где часть значений может быть испорчена или записана в неудобной форме.

Пример:

```sql
TRY_CAST(regexp_replace(NULLIF(trim(age), ''), '\\\\.0+$', '') AS INTEGER)
```

### `NULLIF`

`NULLIF(a, b)` возвращает `NULL`, если `a = b`.
В проекте это основной способ превратить пустую строку в `NULL`.

Пример:

```sql
NULLIF(prod_name, '')
```

### `COALESCE`

`COALESCE(x, y, ...)` возвращает первое не-`NULL` значение.
Это полезно, когда нужно задать запасное значение.

Примеры:

```sql
COALESCE(c.club_member_status, 'unknown')
coalesce(index_name, '')
```

## Работа со строками

### `trim`

Удаляет пробелы по краям строки.
Используется перед очисткой числовых значений, пришедших из CSV.

### `regexp_replace`

Позволяет заменить часть строки по регулярному выражению.
В проекте используется для очистки значений вроде `1.0` перед приведением к целому типу.

Пример:

```sql
regexp_replace(NULLIF(trim(fn), ''), '\\\\.0+$', '')
```

### `regexp_like`

Проверяет, соответствует ли строка регулярному выражению.
Это важная конструкция проекта, потому что с её помощью вычисляются признаки товарных групп.

Пример:

```sql
regexp_like(
  lower(coalesce(index_name, '') || ' ' || coalesce(garment_group_name, '')),
  '(^|[^a-z])(kids|kid|baby|children)([^a-z]|$)'
)
```

### `lower`

Переводит строку в нижний регистр.
Обычно используется вместе с `regexp_like`, чтобы сделать проверку нечувствительной к регистру.

### Конкатенация через `||`

В Trino оператор `||` склеивает строки.
Это удобно, когда нужно анализировать несколько текстовых полей как одну строку.

## Общие табличные конструкции

### `WITH`

`WITH` создаёт временные именованные подзапросы, которые часто называют общими табличными выражениями.
Они делают сложный SQL читаемее и помогают разложить логику по шагам.

Общий вид:

```sql
WITH step_1 AS (...),
step_2 AS (...)
SELECT ...
FROM step_2
```

В проекте `WITH` используется особенно активно:

- в дедупликации размерностей;
- в построении календаря;
- в RFM-витрине;
- в подготовке списков затронутых ключей.

### `JOIN`

`JOIN` объединяет строки из нескольких источников.
В проекте чаще всего используется обычный `JOIN`, то есть внутреннее соединение.

Общий вид:

```sql
FROM fact_table AS f
JOIN dim_table AS d
  ON f.key = d.key
```

Это основа для построения витрин из фактов и измерений.

### `GROUP BY`

`GROUP BY` собирает строки в группы и позволяет вычислять агрегаты на нужном грейне.
Все витрины `mart` опираются именно на эту логику.

Пример:

```sql
GROUP BY sale_date, sales_channel_id
```

## Агрегатные функции

### `count(*)`

Считает количество строк в группе.

### `count(DISTINCT x)`

Считает количество уникальных значений.
В проекте используется, например, для числа покупателей.

### `sum(x)`

Суммирует значения.
Обычно используется для выручки.

### `avg(x)`

Считает среднее значение.

### `min(x)` и `max(x)`

Используются для поиска первой и последней даты покупки.

## Оконные функции

### `row_number() OVER (...)`

Это ключевая конструкция для дедупликации.
Она нумерует строки внутри группы в порядке, заданном `ORDER BY`.

Общий вид:

```sql
row_number() OVER (
  PARTITION BY key
  ORDER BY some_timestamp DESC
)
```

Если после этого выбрать только `WHERE rn = 1`, получится одна “лучшая” строка на ключ.

Именно так в проекте выбираются актуальные версии записей для:

- `dim_article`;
- `dim_customer`.

### `PARTITION BY`

В окне это не физическое партиционирование таблицы.
Здесь оно означает логическое разбиение строк на группы внутри вычисления функции.

Это важное различие:

- `partitioning = ARRAY[...]` в DDL относится к хранению;
- `PARTITION BY` в `OVER (...)` относится к вычислению оконной функции.

## Работа с датами

### `date_trunc`

Обрезает дату до указанной единицы времени.
В проекте чаще всего используется начало месяца.

Пример:

```sql
date_trunc('month', sale_date)
```

### `date_add`

Прибавляет или вычитает временной интервал.

Пример:

```sql
date_add('month', 1, month_start)
date_add('day', -364, snapshot_month)
```

### `date_diff`

Считает разницу между двумя датами в указанной единице времени.

Пример:

```sql
date_diff('day', last_purchase_date, snapshot_month)
```

### `year`, `month`, `day`, `day_of_week`, `week`

Это функции извлечения календарных признаков.
Они используются при построении `dim_date`.

## Генерация последовательностей дат

### `sequence`

Создаёт последовательность значений между двумя границами.
Для дат это удобно при построении календаря.

Пример:

```sql
sequence(min_date, max_date)
```

### `UNNEST`

Преобразует массив или последовательность в набор строк.
Вместе с `sequence` это даёт способ построить таблицу дат.

Пример:

```sql
FROM bounds
CROSS JOIN UNNEST(sequence(min_date, max_date)) AS t(d)
```

## Сравнение и защитные функции

### `least` и `greatest`

`least` возвращает минимальное из значений, `greatest` — максимальное.
В проекте они используются при обновлении агрегата, чтобы:

- не потерять самую раннюю дату покупки;
- корректно обновить самую позднюю дату покупки.

### `NULLIF(..., 0)`

Это важный приём защиты от деления на ноль.

Пример:

```sql
sum(price) / NULLIF(count(DISTINCT customer_id), 0)
```

Если знаменатель равен нулю, выражение вернёт `NULL`, а не ошибку.

### `WHERE NOT EXISTS`

Такой фильтр используется, когда нужно вставить только отсутствующие строки.
В проекте он применяется при пополнении `dim_date`.

## Алиасы

В проекте постоянно используются алиасы таблиц и выражений:

```sql
FROM silver.fact_sales_line AS f
JOIN silver.dim_article AS a
  ON f.article_id = a.article_id
```

Это делает SQL компактнее и проще для чтения.

## Что особенно важно не перепутать

- `PARTITION BY` в оконной функции и `partitioning` в DDL — это разные вещи.
- `CAST` и `TRY_CAST` отличаются поведением при ошибке преобразования.
- `WITH` не создаёт постоянные таблицы, а только временные именованные шаги внутри запроса.
- `MERGE` нужен не для красоты, а для сценария “обновить существующее или вставить новое”.
- `GROUP BY` задаёт грейн результата, поэтому без понимания грейна запрос невозможно интерпретировать правильно.

## Итог

Для этого проекта важнее всего уверенно читать не абстрактный SQL, а конкретный набор конструкций Trino:

- приведение типов и очистку сырых значений;
- дедупликацию через оконные функции;
- работу с датами и месячными границами;
- агрегирование и расчёт метрик;
- обновление данных через `DELETE + INSERT` и `MERGE`.

Если эти конструкции понятны, большая часть SQL-логики проекта становится прозрачной даже без детального чтения каждого файла.
