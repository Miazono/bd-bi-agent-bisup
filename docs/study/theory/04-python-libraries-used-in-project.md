# Python-библиотеки и технические средства, используемые в ETL/ELT-скриптах

## Назначение

Этот документ объясняет не весь Python, а только те библиотеки и приёмы, которые обычно нужны в скриптах загрузки и преобразования данных такого класса.
Цель документа - помочь уверенно читать и писать загрузочные скрипты и скрипты оркестрации, понимать, почему они устроены именно так, и как они обеспечивают повторяемость, идемпотентность и прозрачную эксплуатацию пайплайна.

## Общая картина

В ETL/ELT-скриптах Python обычно решает несколько разных задач:

- получает параметры запуска из командной строки;
- читает конфигурацию и переменные окружения;
- работает с файлами, путями и объектным хранилищем;
- вычисляет хэши и проверяет, не загружали ли файл раньше;
- обращается к HTTP-сервисам, если данные или метаданные приходят по API;
- пишет структурированные логи;
- формирует SQL и отправляет его в Trino через DB API-клиент;
- делает загрузку повторяемой и безопасной при повторном запуске.

В этом проекте набор инструментов именно такой: часть логики опирается на стандартную библиотеку Python, часть - на тонкие внешние клиенты для MinIO и Trino.

## `pathlib` и `os`: работа с путями, средой и файловой системой

### Зачем это нужно

ETL-скрипты почти всегда читают локальные файлы, собирают пути к SQL-артефактам, проверяют наличие исходников и формируют адреса объектов в хранилище.
Здесь очень важно не склеивать строки вручную, потому что это ломает переносимость и делает код хрупким.

### `pathlib`

`pathlib.Path` - современный способ работать с путями в Python.
Он удобнее строк, потому что умеет:

- безопасно склеивать части пути;
- проверять существование файла;
- читать и писать текст;
- получать родительский каталог;
- делать относительные пути.

Пример типичного использования:

```python
from pathlib import Path

ddl_path = Path("sql") / "ddl" / "bronze" / "bronze_hm_articles.sql"
if ddl_path.exists():
    sql_text = ddl_path.read_text(encoding="utf-8")
```

Для ETL-скриптов это важно, потому что SQL-файлы, шаблоны и каталоги с данными часто живут рядом, а путь к ним должен быть выражен явно и читаемо.

### `os`

Модуль `os` в таких скриптах чаще всего используют не для путей, а для:

- чтения переменных окружения через `os.getenv`;
- базовых проверок окружения;
- интеграции с командной строкой и процессом.

Когда задача касается файлов и каталогов, обычно лучше выбирать `pathlib`. Когда нужно быстро и просто прочитать настройку из среды - `os.getenv` остаётся самым прямым вариантом.

## `requests` и `urllib`: HTTP-клиенты и разбор адресов

### Когда нужен HTTP-клиент

HTTP в ETL обычно появляется в трёх случаях:

- источник данных отдаёт API;
- сервис метаданных или оркестрации отдаёт REST-интерфейс;
- скрипт должен проверить состояние внешнего сервиса.

Для такого сценария чаще всего используют `requests`, потому что он делает HTTP-запросы очень удобными: понятные параметры, удобная обработка JSON, простая работа с заголовками и ошибками.

### Когда достаточно `urllib`

Внутри стандартной библиотеки Python есть `urllib`, и его полезно знать хотя бы на базовом уровне.
В ETL-скриптах он часто нужен не как HTTP-клиент, а как набор утилит для разборки адресов.
Например, `urllib.parse.urlparse` помогает понять, есть ли в endpoint схема `http` или `https`, какой у него хост и порт.

Это полезно для MinIO и Trino, где endpoint может приходить в разных формах:

- `http://localhost:9000`;
- `http://minio:9000`;
- `localhost:9000`.

В таком случае скрипт должен не гадать, а разобрать адрес и выбрать корректный режим соединения.

### Практический вывод

Если в пайплайне есть настоящий HTTP API, `requests` почти всегда удобнее.
Если нужен только разбор адреса или минимальная зависимость, `urllib.parse` из стандартной библиотеки даёт достаточно.
В этом проекте как раз важен второй сценарий: endpoint нужно корректно разобрать и передать клиенту хранилища или Trino.

## `hashlib`: контроль изменений и идемпотентность

### Что это такое

`hashlib` предоставляет криптографические хэши, например MD5, SHA-1, SHA-256.
В ETL MD5 используют не для безопасности, а как компактный отпечаток файла или набора данных.

### Зачем это нужно в пайплайне

Хэш позволяет быстро ответить на вопрос: изменился ли входной файл с прошлого запуска?
Если хэш совпадает, можно не загружать файл повторно и не плодить дубли.
Это один из простых и надёжных способов сделать raw-загрузку идемпотентной.

Типовой приём:

```python
import hashlib

def compute_md5(file_path: str) -> str:
    hasher = hashlib.md5()
    with open(file_path, "rb") as file_obj:
        for chunk in iter(lambda: file_obj.read(1024 * 1024), b""):
            hasher.update(chunk)
    return hasher.hexdigest()
```

Важно читать файл порциями, а не целиком, если он может быть большим.
Так скрипт не тратит лишнюю память и остаётся устойчивым на более крупных объёмах.

### Как это работает в загрузках

Обычно паттерн такой:

- считаем локальный MD5;
- читаем хэш уже загруженного объекта или метаданных;
- если хэши совпали, пропускаем загрузку;
- если нет, перезаписываем объект и обновляем метаданные.

Это уменьшает лишние записи в объектное хранилище и делает повторный запуск безопасным.

## `json` и `csv`: структурированные форматы данных

### `csv`

Для табличных исходников CSV - один из самых распространённых форматов на входе в ETL.
Модуль `csv` полезен, когда нужно:

- читать CSV без внешних зависимостей;
- контролировать разделители, кавычки и экранирование;
- аккуратно писать производные CSV;
- не полагаться целиком на поведение внешнего клиента.

Даже если сам пайплайн дальше передаёт CSV в объектное хранилище как файл, полезно помнить, что CSV - это не просто текст, а формат с правилами разделения и экранирования.

### `json`

`json` нужен, когда данные или конфигурация приходят в JSON-формате.
В ETL он часто используется для:

- ответов API;
- метаданных;
- схем и описаний моделей;
- промежуточных конфигураций;
- тестовых фикстур.

Если задача требует только простого хранения табличного файла, JSON не нужен.
Но как только скрипт начинает разговаривать с сервисом по API или хранить структурированную конфигурацию, `json` становится базовым инструментом.

### Практический выбор

- CSV - для строковых табличных данных и raw-слоя.
- JSON - для структурированных метаданных, настроек и ответов сервисов.

В хороших пайплайнах эти форматы не смешивают без необходимости: CSV остаётся форматом данных, JSON - форматом метаданных и обмена.

## `argparse` и `click`: интерфейс запуска скриптов

### `argparse`

`argparse` - стандартный способ описать параметры командной строки.
Он особенно уместен в ETL-скриптах, потому что:

- не требует внешних зависимостей;
- хорошо подходит для простых batch-утилит;
- позволяет задавать значения по умолчанию и справку;
- делает запуск скрипта самодокументируемым.

Пример типичного сценария:

```python
import argparse

parser = argparse.ArgumentParser(description="Load raw files to MinIO")
parser.add_argument("--load-date", default="2026-03-19")
parser.add_argument("--source-dir", required=True)
args = parser.parse_args()
```

Для ingestion-скриптов это особенно полезно, потому что запуск часто зависит от даты батча, каталога с исходниками или режима окружения.

### `click`

`click` удобен, когда CLI становится сложнее:

- много команд;
- вложенные подкоманды;
- красивый вывод;
- больше интерактивности;
- нужен более декларативный стиль.

Для небольших ETL-утилит `argparse` обычно достаточно и даже предпочтительнее, потому что он встроен в Python и проще для сопровождения.
`click` стоит выбирать тогда, когда интерфейс запуска действительно разрастается.

### Практический вывод

В простых загрузчиках лучше начинать с `argparse`.
Если же скрипт превращается в полноценный CLI-инструмент для нескольких операций, `click` может дать более удобную структуру.

## `logging`: наблюдаемость и эксплуатация

### Зачем нужен логгер

ETL-скрипт редко работает "на глаз".
Его запускают по расписанию, в контейнере или вручную, а потом по логам понимают, что произошло:

- какой батч стартовал;
- какой файл обработан;
- что было пропущено как уже загруженное;
- где произошла ошибка;
- сколько строк попало в таблицу.

Поэтому вместо `print()` лучше использовать `logging`.

### Базовый шаблон

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s | %(levelname)s | %(message)s",
)
logger = logging.getLogger(__name__)

logger.info("Load started")
logger.warning("File is missing")
logger.error("Unexpected failure")
```

### Почему это важно именно для пайплайнов

Логи делают процесс наблюдаемым и помогают в трёх вещах:

- диагностировать сбои;
- подтверждать, что повторный запуск не создал дублей;
- отслеживать объём загруженных данных.

Для batch-процессов это часто важнее красивого пользовательского интерфейса.

## `datetime`: даты, батчи и временные срезы

### Что обычно делают с датами

ETL-скрипты почти всегда завязаны на даты:

- имя папки raw-слоя содержит дату загрузки;
- часть таблиц партиционируется по месяцу;
- при инкрементальной обработке нужно найти затронутые периоды;
- отчётные витрины строятся по календарным срезам.

Для этого используют `datetime.date`, `datetime.datetime`, `date.fromisoformat`, `date.today()` и преобразования к ISO-строке.

### Почему важен ISO-формат

Формат `YYYY-MM-DD` удобен потому, что:

- его понимают люди;
- его легко сравнивать;
- его легко хранить в путях и параметрах;
- он хорошо подходит для партиций и батчей.

Типичный пример:

```python
from datetime import date

load_date = date.today().isoformat()
```

### Практический смысл

В пайплайне дата - это не просто справочная информация.
Это часть ключа загрузки, часть структуры пути и часть логики идемпотентного пересчёта.

## `typing` и `dataclasses`: понятные структуры данных

### `typing`

`typing` помогает явно описывать ожидаемые типы:

- `list[str]`;
- `dict[str, str]`;
- `Optional[str]`;
- `tuple[str, str, str]`.

Это особенно полезно в ETL-скриптах, где много промежуточных значений, словарей конфигурации и результатов запросов.
Аннотации типов снижают шанс перепутать путь, SQL-строку, имя таблицы или дату батча.

### `dataclasses`

`dataclass` удобно использовать для простых объектов-носителей данных:

- информация об объекте в хранилище;
- описание таблицы;
- набор настроек;
- результат парсинга.

Пример:

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class S3ObjectInfo:
    key: str
    size: int
    etag: str | None = None
```

Почему это полезно:

- код становится короче;
- структура данных видна сразу;
- объект можно передавать между функциями без ручного словаря;
- `frozen=True` делает его неизменяемым, что удобно для служебных описаний.

### Когда это особенно уместно

Если скрипт начинает разрастаться, словари с "магическими" ключами быстро становятся источником ошибок.
`dataclass` и аннотации типов помогают держать такие места под контролем.

## Переменные окружения и конфигурация

### Зачем вообще выносить настройки в окружение

ETL/ELT-скрипты почти всегда должны запускаться в разных средах:

- локально;
- в Docker;
- в CI/CD;
- на сервере по расписанию.

Чтобы один и тот же код работал в разных местах, параметры подключения и пути выносят в переменные окружения.
Это позволяет не хардкодить:

- адрес MinIO;
- учётные данные;
- хост Trino;
- схему по умолчанию;
- префиксы слоёв;
- внешние endpoint'ы.

### Типовой подход

Часто используется связка:

- `.env`-файлы для локальной разработки;
- `load_dotenv()` для подхвата значений;
- единый объект настроек, собранный из среды;
- валидация обязательных параметров на старте.

Такой подход особенно полезен для повторяемости: если параметр не задан, скрипт должен упасть сразу и понятным сообщением, а не через десять минут работы.

### Почему это важно для идемпотентности

Когда конфигурация управляется явно, проще гарантировать, что:

- один и тот же батч попадёт в одну и ту же папку;
- один и тот же Trino-каталог будет использоваться для одной и той же среды;
- повторный запуск не "поедет" в другую систему случайно.

Идемпотентность начинается не только с SQL-логики, но и с корректной конфигурации запуска.

## Trino-клиенты и концепция DB API

### Что такое DB API

DB API в Python - это общий интерфейс для работы с SQL-базами через единый стиль вызовов.
Идея простая:

- открыть соединение;
- получить cursor;
- выполнить SQL;
- забрать строки результата;
- закрыть cursor.

Точный набор классов и методов зависит от драйвера, но логика почти всегда одна и та же.

### Почему это удобно

Когда код пишет SQL в Trino через DB API, он остаётся предсказуемым:

- запрос можно выполнять строкой;
- результат можно читать как список кортежей;
- одну и ту же обвязку легко переиспользовать;
- бизнес-логика не смешивается с сетевым протоколом.

### Trino-клиент

Для Trino обычно используют клиент, который предоставляет DB API-подобный интерфейс.
Это важно, потому что Trino в ETL-проектах часто выступает как единая SQL-точка входа к raw, bronze, silver и mart-слоям.

Типовая схема работы:

```python
from trino.dbapi import connect

conn = connect(
    host="trino",
    port=8080,
    user="example",
    catalog="iceberg",
    schema="default",
    http_scheme="http",
)

cur = conn.cursor()
cur.execute("SELECT 1")
rows = cur.fetchall()
cur.close()
```

### Что важно помнить

- `execute()` запускает SQL, но не обязан возвращать строки.
- `fetchall()` и `fetchone()` нужны для чтения результата.
- cursor лучше закрывать после использования.
- SQL для загрузки обычно формируется отдельно, а клиент только выполняет его.

### Почему это особенно важно в пайплайне

Trino в таком проекте - не просто база для запросов, а исполнительный слой пайплайна.
Через него создаются схемы, загружаются таблицы, удаляются старые батчи и пересобираются витрины.

## Идемпотентные загрузки: главный практический принцип

### Что означает идемпотентность

Идемпотентная загрузка - это загрузка, которую можно безопасно запускать повторно, не получая нежелательных дублей и неконсистентных результатов.
Для data engineering это один из базовых принципов качества.

### Типовые приёмы

- проверка существования файла или объекта перед загрузкой;
- сравнение MD5 или другого хэша;
- `CREATE SCHEMA IF NOT EXISTS` и `DROP TABLE IF EXISTS`;
- `DELETE` по `batch_id` перед повторной вставкой;
- пересборка только затронутого месяца или среза;
- логирование количества строк после загрузки;
- проверка, что батч действительно полный.

### Почему это работает

Каждый из этих шагов снижает риск того, что повторный запуск:

- создаст дубли;
- оставит частично обновлённые таблицы;
- смешает данные разных батчей;
- загрузит уже обработанный объект ещё раз.

### Хорошая ментальная модель

Идемпотентность в ETL стоит воспринимать как сочетание трёх вещей:

- контроль входа;
- контроль состояния хранилища;
- контроль результата после выполнения.

Если все три уровня покрыты, повторный запуск становится штатным сценарием, а не аварией.

## Как это складывается в рабочий пайплайн

Типичный ETL/ELT-скрипт такого класса использует эти средства примерно так:

1. Читает параметры запуска через `argparse` или конфигурацию из окружения.
1. Разбирает endpoint'ы через `urllib.parse`.
1. Находит локальные файлы через `pathlib`.
1. Проверяет содержимое через `hashlib` или наличием объекта.
1. Загружает raw-файлы в объектное хранилище.
1. Передаёт SQL в Trino через DB API-клиент.
1. Логирует шаги через `logging`.
1. Опирается на `datetime` для батчей и дат.
1. Использует `typing` и `dataclasses` для понятных структур и сигнатур.
1. Повторный запуск делает безопасным за счёт проверок и удаления старых срезов.

## Короткий вывод

Для ETL/ELT-скриптов важны не столько сами библиотеки по отдельности, сколько их связка.
`pathlib`, `os`, `hashlib`, `logging`, `datetime`, `typing`, `dataclasses`, `argparse`, `urllib.parse` и DB API-клиенты Trino образуют минимальный практический набор, который позволяет писать понятные, переносимые и идемпотентные загрузки.

Если запомнить одну мысль, то она такая: хороший data pipeline - это не только SQL и не только Python, а аккуратно собранный процесс, где каждая библиотека отвечает за свой маленький, но важный участок надёжности.
