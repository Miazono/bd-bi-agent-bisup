# Разбор контура оценки

## Назначение документа

Этот документ разбирает, как именно в проекте устроена проверка BI-части: какие файлы участвуют, что реально проверяется, какие данные для этого используются и где у текущей схемы есть ограничения.
Здесь важно смотреть только на фактические файлы репозитория, потому что именно контур оценки чаще всего выглядит “солиднее”, чем есть на самом деле.

## Из каких файлов состоит контур

В текущем состоянии контур оценки в `bi-agent/eval/` опирается на четыре артефакта:

- `questions.json`
- `run_eval.py`
- `golden.sql`
- `llm_judge.py`

Дополнительно в качестве очень базовой проверки используется `tests/test_bi_agent.py`, но это уже не полноценный контур оценки, а только минимальная проверка вокруг HTTP-слоя.

## `questions.json`

### Что хранится в файле

`questions.json` содержит массив тестовых сценариев.
Каждый сценарий включает:

- `id` — идентификатор вопроса;
- `question` — текст вопроса на естественном языке;
- `target_mart` — ожидаемая витрина, на которую должен опереться запрос;
- `expected_columns` — ожидаемые колонки результата.

В файле 30 вопросов, и они распределены по пяти витринам:

- `mart.sales_daily_channel`
- `mart.sales_monthly_category`
- `mart.customer_segment_monthly`
- `mart.repeat_purchase_category`
- `mart.customer_rfm_monthly`

### Что это означает

Файл уже задаёт не просто набор вопросов, а целевую структуру проверки.
По замыслу тут есть как минимум две оси контроля:

- правильность выбора витрины;
- соответствие результата ожидаемым колонкам.

Но важно понимать, что сам по себе `questions.json` ничего не проверяет. Он только хранит сценарии для внешнего скрипта.

## `run_eval.py`

### Общая роль

`run_eval.py` — это минимальный автоматический запуск оценочного набора.
Он делает четыре вещи:

1. читает `questions.json`;
2. собирает URL API WrenAI;
3. отправляет каждый вопрос в сервис;
4. проверяет, есть ли в сгенерированном SQL имя ожидаемой витрины.

### Как устроены основные функции

#### `load_questions(path)`

Читает JSON-файл и возвращает список вопросов.
Логика простая: файл открывается в кодировке UTF-8, затем вызывается `json.load`.

#### `build_api_url()`

Формирует конечный URL для запроса к WrenAI.
Здесь используются две переменные окружения:

- `WREN_AI_ENDPOINT` — обязательная;
- `WREN_AI_QUERY_PATH` — необязательная, по умолчанию `/api/v1/query`.

Функция аккуратно склеивает базовый адрес и путь, учитывая слэши на стыке.
Если `WREN_AI_ENDPOINT` не задан, скрипт завершает работу с ошибкой.

#### `extract_sql(response_json)`

Ищет SQL в ответе API.
Функция смотрит:

- в корневые ключи `sql`, `query`, `generated_sql`;
- затем в вложенные контейнеры `data`, `result`, `payload`.

Это полезная эвристика, потому что разные версии API могут возвращать SQL в разной структуре.
Но это именно эвристика, а не строгий контракт.

#### `request_sql(api_url, question_text, timeout=30)`

Формирует POST-запрос с JSON-полем `question`, отправляет его в WrenAI и пытается извлечь SQL из ответа.
Для запроса используется `urllib.request.Request`, а сам ответ читается через `urlopen`.

#### `main()`

Основной цикл проходит по всем вопросам и для каждого сценария делает следующее:

- отправляет вопрос в API;
- ловит `HTTPError`, `URLError` и `json.JSONDecodeError`;
- проверяет, что SQL вообще получен;
- проверяет, что в SQL встречается `target_mart` без учёта регистра;
- пишет `pass` или `fail` в лог.

### Что реально проверяется

Сейчас проверка очень узкая.
Фактически скрипт отвечает только на вопрос: присутствует ли имя целевой витрины в тексте SQL.

Это означает, что скрипт проверяет:

- факт получения SQL;
- факт выбора хотя бы нужной витрины в строке SQL.

### Что не проверяется

Скрипт не проверяет:

- корректность синтаксиса SQL;
- выполнение SQL на движке;
- правильность `JOIN`;
- правильность фильтров;
- правильность агрегаций;
- совпадение с `expected_columns`;
- совпадение с реальным результатом на данных;
- наличие лишних таблиц или лишних полей;
- качество ответа на естественном языке.

То есть это не полноценная оценка качества NL2SQL, а только минимальная строковая проверка, что запрос хоть как-то связан с нужной витриной.

## `golden.sql`

Файл `bi-agent/eval/golden.sql` сейчас пустой.

Это важная деталь: в репозитории есть название, которое намекает на эталонные SQL-запросы, но фактического содержимого пока нет.
Следовательно, сравнение с золотым эталоном на текущий момент не реализовано.

Если говорить строго, этот файл сейчас выполняет роль заготовки под будущую более серьёзную систему оценки.

## `llm_judge.py`

Файл `bi-agent/eval/llm_judge.py` тоже пустой.

По названию можно ожидать логику, где отдельная языковая модель оценивает качество ответа, сравнивает SQL, интерпретирует результат или выносит суждение о соответствии вопросу.
Но фактически такая логика пока не написана.

Это значит, что судящий слой в проекте пока отсутствует как реализованный механизм.

## Роль `tests/test_bi_agent.py`

Тесты вокруг BI-агента очень скромные и не заменяют контур оценки.
В `tests/test_bi_agent.py` есть только две проверки:

- что эндпоинт `/health` отвечает `200`;
- что POST-запрос к `/api/v1/query` возвращает JSON, где поле `sql` содержит подстроку `select`.

### Что эти тесты дают

Они подтверждают:

- что базовый HTTP-контур доступен;
- что запрос доходит до API;
- что в ответе есть хотя бы какой-то SQL-подобный результат.

### Чего они не дают

Эти тесты не показывают:

- что BI-агент выбирает правильную витрину;
- что запрос правильно интерпретирует вопрос;
- что SQL соответствует семантическому слою;
- что evaluation на 30 вопросах вообще проходит;
- что результат можно считать качественным с точки зрения аналитики.

## Ограничения текущего контура оценки

Если коротко, текущая схема ограничена сразу в нескольких местах:

- оценка построена на строковом поиске названия витрины;
- `expected_columns` пока не используются;
- эталонные SQL-запросы не заполнены;
- отдельный LLM-judge не реализован;
- тесты проверяют только минимальное поведение HTTP-слоя.

Из-за этого контур нельзя считать полноценной системой валидации качества BI-агента.
Скорее это рабочая заготовка, которая позволяет не потерять сценарии и не сломать совсем базовое взаимодействие с сервисом.

## Что должно появиться в целевом варианте

Чтобы контур оценки стал действительно полезным, в нём должны появиться как минимум такие элементы:

- проверка соответствия ответа заданному `target_mart`;
- использование `expected_columns` не только как справки, но и как критерия;
- эталонные SQL-артефакты для сравнения;
- отдельный механизм оценки качества ответа, если строковой проверки недостаточно;
- более ясная граница между тестом доступности сервиса и оценкой качества NL2SQL.

Тогда контур оценки можно будет использовать не только как формальную проверку, но и как инструмент для понимания, насколько BI-часть проекта действительно работает в нужном направлении.
