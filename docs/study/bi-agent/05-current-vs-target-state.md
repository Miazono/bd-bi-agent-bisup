# Текущее и целевое состояние BI-части

## Назначение документа

Этот документ нужен, чтобы без украшений зафиксировать, что именно уже существует в BI-контуре проекта `db-bi-agent-bisup`, а что пока представляет собой заготовку, каркас или незавершённое направление.
Для защиты проекта это особенно важно: по BI-части легко создать впечатление законченной системы, хотя по факту часть элементов реализована полностью, часть только сконфигурирована, а часть пока сведена к пустым файлам.

## Что входит в BI-контур проекта

В репозитории BI-часть собрана вокруг `bi-agent/` и `infra/wrenai/`.
Внутри этого контура сейчас есть несколько разных уровней:

- семантический слой, который описывает, какие таблицы и поля можно показывать BI-агенту;
- конфигурация WrenAI, которая задаёт модели, движки, хранилище документов и пайплайн обработки;
- оценочный контур, который хранит набор вопросов и скрипт для проверки генерации SQL;
- тесты, которые проверяют только минимальную доступность HTTP-слоя и базовый ответ на запрос SQL;
- документация по данным и витринам, на которую должна опираться BI-экспозиция.

То есть в проекте уже есть не просто набор файлов, а целая цепочка артефактов вокруг NL2SQL-сценария. При этом степень готовности у этих артефактов разная.

## Текущее состояние

### 1. Семантический слой существует и ориентирован на marts

Фактическая BI-экспозиция сейчас описана в `bi-agent/semantic_layer/`.
Там есть пять YAML-файлов с описаниями витрин:

- `mart_sales_daily_channel.yaml`
- `mart_sales_monthly_category.yaml`
- `mart_customer_segment_monthly.yaml`
- `mart_repeat_purchase_category.yaml`
- `mart_customer_rfm_monthly.yaml`

Каждый файл задаёт:

- имя сущности;
- ссылку на физическую таблицу в схеме `mart`;
- описание;
- грейн;
- список колонок с типами и текстовыми пояснениями.

Это важный признак зрелости BI-части: агенту не показывают сырые таблицы как основной слой для общения с пользователем, а выводят на витрины, то есть на более стабильный и понятный аналитический уровень.

При этом `bi-agent/AGENTS.md` прямо задаёт правило: основная экспозиция должна строиться на `marts`, а `silver`-таблицы можно показывать только выборочно, если нужно объяснение или более продвинутое моделирование. `bronze`-таблицы как основные BI-сущности использовать нельзя.

### 2. Конфигурация WrenAI заполнена

Файл `infra/wrenai/config.yaml` уже задаёт рабочую конфигурацию контура:

- LLM-провайдер `litellm_llm` с моделью `gpt-4o`;
- embedder `litellm_embedder` с моделью `text-embedding-3-small`;
- engine `wren_ui` на `http://wren-ui:3000`;
- engine `wren_ibis` на `http://ibis-server:8000`;
- документное хранилище `qdrant` на `http://qdrant:6333`;
- пайплайн из множества шагов: индексация схем, поиск, генерация SQL, коррекция SQL, диагностика, генерация follow-up, ответы, рекомендации, извлечение функций, извлечение таблиц и другие.

Это означает, что на уровне инфраструктуры BI-контур не пустой: он уже подготовлен для генерации и обработки запросов на естественном языке.

### 3. Оценочный контур есть, но он упрощён

В `bi-agent/eval/` уже лежат:

- `questions.json` с набором проверочных вопросов;
- `run_eval.py` со скриптом запуска проверки;
- `golden.sql`;
- `llm_judge.py`.

Но фактическая степень реализации тут разная:

- `questions.json` заполнен и содержит 30 вопросов;
- `run_eval.py` работает как минимальная автоматическая проверка;
- `golden.sql` пустой;
- `llm_judge.py` пустой.

То есть оценочный контур пока существует скорее как каркас, чем как полноценная система сравнения качества.

### 4. Тесты очень узкие

В `tests/test_bi_agent.py` есть только две проверки:

- проверка, что `/health` отвечает статусом `200`;
- проверка, что POST-запрос к `/api/v1/query` возвращает JSON с полем `sql`, в котором есть слово `select`.

Это полезно как smoke test, но этого недостаточно, чтобы подтвердить корректность генерации SQL, выбор правильной витрины, соответствие колонок или устойчивость к неоднозначным формулировкам.

## Что это означает по сути

Текущий BI-контур можно описать так:

1. Есть целевой способ работы через витрины `mart`.
2. Есть описания этих витрин в семантическом слое.
3. Есть WrenAI-конфигурация для генерации SQL по вопросам на естественном языке.
4. Есть минимальная проверка, что сервис отвечает и возвращает SQL.
5. Но пока нет полноценной автоматической валидации результата и нет отдельного judge-слоя.

Иначе говоря, проект уже показывает архитектурное направление BI-части, но не доводит его до полностью зрелого оценочного и контрольного контура.

## Целевое состояние

Целевое состояние BI-части можно сформулировать как более строгую и проверяемую версию текущего контура.

### 1. Семантический слой должен стать главным интерфейсом к данным

В целевом варианте BI-агент должен опираться не на произвольные таблицы, а на явно описанные аналитические сущности.
Для этого нужны:

- заполненные semantic layer-файлы;
- согласованные описания грейна;
- согласованные типы колонок;
- согласованные текстовые определения метрик и смыслов полей;
- единая связь между витринами, именами сущностей и пользовательскими запросами.

### 2. Промпты и поведение агента должны быть согласованы с витринами

Целевое состояние предполагает, что агент:

- сначала ищет ответ среди `mart`;
- при необходимости может использовать `silver` как вспомогательный уровень;
- не строит основную логику на `bronze`;
- генерирует SQL, который соответствует документированным полям и грейну.

### 3. Оценка должна проверять не только наличие SQL

В текущем виде система проверки слишком слабая.
В целевом состоянии она должна проверять хотя бы несколько разных аспектов:

- выбран ли правильный `mart`;
- соответствует ли SQL ожидаемым колонкам;
- соответствует ли SQL смыслу вопроса;
- не использует ли запрос лишние сущности;
- можно ли воспроизвести результат на фиксированном наборе сценариев.

### 4. Нужен разделённый контроль качества

Целевой контур BI лучше строить в нескольких слоях:

- синтаксическая проверка SQL;
- проверка выбранной витрины;
- проверка соответствия ожидаемым полям;
- при необходимости оценка качества ответа отдельным судящим слоем.

Такой подход делает BI-часть не просто “генератором SQL”, а контролируемым решением, где можно объяснить, почему запрос считается удачным или неудачным.

## Что уже можно честно говорить на защите

Если описывать проект строго по фактам, то корректная формулировка будет такой:

- BI-контур в проекте уже спроектирован вокруг витрин и семантического слоя;
- `WrenAI` настроен как средство NL2SQL-поиска по этим витринам;
- часть проверки качества реализована только в минимальном виде;
- целевая схема понятна, но не все её элементы пока наполнены содержанием.

Это не слабость документации, а нормальное и честное описание текущего состояния.
