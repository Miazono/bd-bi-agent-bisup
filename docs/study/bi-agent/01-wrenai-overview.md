# WrenAI и BI-контур проекта

## Что такое WrenAI

WrenAI в этом проекте — это внешний BI-слой, который помогает строить SQL-ответы на вопросы на естественном языке.
Он не хранит данные сам по себе и не заменяет слой lakehouse.
Его задача — использовать уже опубликованные витрины и преобразовать запрос пользователя в SQL, который можно выполнить поверх Trino.

## Место в проекте

BI-контур находится над слоем `marts`.
Это важно, потому что в проекте уже есть полноценная аналитическая публикация:

- `mart.sales_daily_channel`
- `mart.sales_monthly_category`
- `mart.customer_segment_monthly`
- `mart.repeat_purchase_category`
- `mart.customer_rfm_monthly`

С точки зрения архитектуры это означает, что WrenAI должен работать не с сырыми таблицами и не с `bronze`, а с уже подготовленными витринами.

## Компоненты BI-контура

В текущем репозитории BI-часть состоит из нескольких сущностей.

### `wren-ui`

Это интерфейс, через который пользователь взаимодействует с BI-системой.

### `wren-ai-service`

Это сервис, который обрабатывает запросы, использует конфигурацию пайплайнов и взаимодействует с векторным хранилищем.

### `wren-engine`

Это внутренний движок Wren-контурa, который помогает обслуживать SQL-цепочки и внутренние функции системы.

### `ibis-server`

Это отдельный движок Wren, используемый для части SQL-функциональности.

### `qdrant`

Это векторное хранилище, куда попадают индексы схем, инструкций, вопросов и SQL-пар.

### `bootstrap`

Это стартовый контейнер, который подготавливает данные для Wren-контура.

## Что делает BI-часть в текущем виде

Текущая реализация BI-контура решает несколько задач:

- индексирует схему и описания данных;
- хранит полезный контекст в Qdrant;
- пытается выбрать нужную витрину по вопросу пользователя;
- генерирует SQL;
- корректирует SQL;
- позволяет запускать базовую проверку вопросов;
- служит заготовкой для дальнейшего расширения семантического слоя.

## Чего BI-часть не делает сама по себе

WrenAI не заменяет физическую модель данных.
Он не создаёт таблицы lakehouse и не управляет загрузкой `raw`, `bronze` или `silver`.

Он также не должен быть источником истины для бизнес-логики.
Источник истины здесь остаются:

- DDL;
- SQL-запросы;
- `docs/data/schema.md`;
- `docs/data/marts.md`.

## Почему это важно для защиты

На защите полезно уметь разделять три уровня:

- физические данные и витрины;
- семантическое описание;
- BI-агент, который делает вывод SQL на основе этого описания.

Если эти уровни смешать, будет трудно объяснить, что именно уже реализовано, а что ещё только подготавливается.
