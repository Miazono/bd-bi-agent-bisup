datalakehouse-bi-agent/
├── AGENTS.md
│   Главные инструкции для Codex на уровне всего репозитория.
│   Здесь описываются цель проекта, общие правила изменений, базовые команды и ограничения.
│
├── ARCHITECTURE.md
│   Краткое описание целевой архитектуры проекта.
│   Это не операционная инструкция, а схема того, как связаны MinIO, Iceberg, Trino, Hive Metastore и BI-агент.
│
├── README.md
│   Входная точка для человека.
│   Должен кратко отвечать на вопросы: что это за проект, какой стек, какие этапы конвейера, как запускать.
│
├── .env.example
│   Пример переменных окружения без секретов.
│   Нужен как шаблон для локального `.env`.
│
├── .gitignore
│   Правила исключения временных файлов, локальных данных, секретов и больших артефактов.
│
├── notes.md
│   Рабочие заметки по проекту.
│   Не является источником истины для архитектуры, но полезен для набросков решений и TODO.
│
├── infra/
│   Всё, что относится к локальному разворачиванию стека.
│   Здесь лежат compose-файлы и конфиги сервисов.
│
│   ├── AGENTS.md
│   │   Инструкции для Codex по инфраструктурным изменениям.
│   │   Что можно менять без подтверждения, а что нет.
│   │
│   ├── docker-compose.yml
│   │   Локальное описание сервисов проекта.
│   │   На раннем этапе может быть частично заполненным, но в целевой версии должен поднимать MinIO,
│   │   Hive Metastore, metastore DB, Trino и BI-агент.
│   │
│   ├── hive/
│   │   Конфиги Hive Metastore.
│   │   Здесь должен лежать `hive-site.xml` и всё, что нужно именно metastore-сервису.
│   │
│   ├── minio/
│   │   Конфиги и скрипты инициализации MinIO.
│   │   Обычно сюда кладут init-скрипт для создания бакета `lakehouse` и служебных префиксов слоёв.
│   │
│   ├── trino/
│   │   Конфиги Trino.
│   │   Здесь должны лежать базовый config, при необходимости jvm/node settings и каталог(и) connector'ов.
│   │
│   └── wrenai/
│       Конфиги BI-агента.
│       Здесь должен лежать config подключения к Trino и LLM-провайдеру.
│
├── ingestion/
│   Скрипты загрузки и подготовки данных по слоям.
│   Это технический конвейер между raw-файлами и аналитическими таблицами.
│
│   ├── AGENTS.md
│   │   Локальные правила для ingestion-слоя.
│   │   Какие слои строим, как именуем таблицы, что считать допустимым изменением схемы.
│   │
│   ├── load_raw.py
│   │   Загрузка исходных файлов в MinIO как есть.
│   │   Это этап сохранения исходных файлов без бизнес-трансформаций.
│   │
│   ├── load_bronze.py / load_iceberg.py
│   │   Первый этап записи в lakehouse-таблицы.
│   │   Если остаёшься на medallion-подходе, лучше использовать имя `load_bronze.py`.
│   │
│   ├── load_silver.py
│   │   Построение очищенных и согласованных таблиц предметной области.
│   │   Здесь уже допустимы очистка, дедупликация, стандартизация типов, обогащение.
│   │
│   └── utils/
│       Общие утилиты для ingestion.
│       Здесь должны лежать клиенты S3/Trino, функции чтения схем, логирование, общие вспомогательные функции.
│
├── sql/
│   SQL-витрины верхнего аналитического слоя.
│   Это конечные таблицы или представления, на которые будет опираться BI-агент.
│
│   ├── AGENTS.md
│   │   Локальные инструкции для работы с витринами.
│   │   Например: одна витрина — один бизнес-вопрос, фиксированный grain, имена в формате mart_*.
│   │
│   └── *.sql
│       Отдельные SQL-файлы для создания витрин.
│       Даже если файлов пока нет, папка уже правильно выделена.
│
├── bi-agent/
│   Всё, что относится к NL2SQL и качественному циклу BI-агента.
│
│   ├── AGENTS.md
│   │   Инструкции для Codex по семантическому слою, промптам и оценке качества.
│   │
│   ├── semantic_layer/
│   │   Описание бизнес-моделей, сущностей, связей и метрик для BI-агента.
│   │   Главная цель этой папки — сделать схему данных понятной LLM.
│   │
│   ├── prompts/
│   │   Системный промпт и связанные промпты для работы агента.
│   │   Это место для управляемой логики промптов, а не для случайных черновиков.
│   │
│   └── eval/
│       Тестовые вопросы, эталонные SQL и, при необходимости, LLM-as-a-Judge.
│       Нужен для оценки качества NL2SQL.
│
├── docs/
│   Человеческая документация проекта.
│   Это не код и не конфиг, а объяснение решений и устройства проекта.
│
│   ├── data/
│   │   Документация по данным и витринам.
│   │
│   │   ├── schema.md
│   │   │   Человекочитаемое описание схем таблиц, grain и назначения слоёв.
│   │   │   Этот файл можно редактировать вручную.
│   │   │
│   │   ├── catalog_generated.md
│   │   │   Производный каталог физических таблиц и столбцов.
│   │   │   Обновляется скриптом `scripts/gen_schema.py`.
│   │   │
│   │   └── marts.md
│   │       Краткое описание витрин: что считает каждая витрина, какие вопросы закрывает, какие метрики содержит.
│   │
│   ├── decisions/
│   │   ADR и архитектурные решения.
│   │   Здесь стоит фиксировать спорные выборы: почему Iceberg, почему Trino, почему WrenAI.
│   │
│   ├── prompts/
│   │   Документация по инженерии промптов.
│   │   Что пробовали, что сработало, что ухудшило результаты.
│   │
│   └── setup/
│       Инструкции по локальному запуску и обзору стека.
│       Здесь удобно держать `local-setup.md` и `stack-overview.md`.
│
├── scripts/
│   Одноразовые или вспомогательные служебные скрипты, не являющиеся основным конвейером.
│
│   └── gen_schema.py
│       Генерация каталога физических таблиц по DDL-файлам репозитория.
│       Это хороший кандидат на авто-генерируемый артефакт.
│
└── tests/
    Автотесты по основным слоям проекта.

    ├── test_ingestion.py
    │   Проверка ingestion-логики и инвариантов загрузки.
    │
    ├── test_marts.py
    │   Проверка витрин и их бизнес-инвариантов.
    │
    └── test_bi_agent.py
        Проверка BI-агента: хотя бы smoke-тесты по семантическому слою и генерации SQL.
